## REQ_USER_001 — Просмотр пользователей (ADMIN/SELLER)

1. Пользователи с ролями "ADMIN" и "SELLER" должны иметь доступ к просмотру списка пользователей.
2. В списке пользователей для каждой записи должны отображаться:
   - идентификатор пользователя;
   - логин;
   - роль пользователя (например, ADMIN, SELLER, USER);
4. Обычный пользователь (роль USER) не должен иметь доступа к просмотру списка пользователей.

---

## REQ_USER_002 — Создание пользователя (ADMIN)

1. Пользователь с ролью "ADMIN" должен иметь возможность создавать новых пользователей.
2. При создании пользователя обязательными для заполнения являются:
   - логин (уникальный идентификатор учётной записи);
   - пароль (в соответствии с REQ_AUTH_002 — Требования к паролю, за исключением случаев, описанных в REQ_USER_006);
   - роль пользователя (ADMIN, SELLER, USER).
3. При попытке создать пользователя с уже существующим логином система должна:
   - не создавать пользователя;
   - отобразить сообщение об ошибке о неуникальном логине.
4. После успешного создания пользователь должен иметь возможность авторизоваться с заданными логином и паролем.
5. Доступ к операции создания пользователей должен быть ограничен только пользователями с ролью "ADMIN"; пользователи с другими ролями не должны видеть элементы управления для создания пользователя.

---

## REQ_USER_003 — Редактирование пользователя (ADMIN)

1. Пользователь с ролью "ADMIN" должен иметь возможность изменять данные профиля других пользователей.
2. Доступные для редактирования поля:
   - логин;
   - пароль (с возможностью задать значение, не соответствующее REQ_AUTH_002, в соответствии с REQ_USER_006);
   - роль.
3. Логин пользователя:
   - является изменяемым при условии проверки уникальности (не должен совпадать с логином другого пользователя).
4. При изменении роли или статуса пользователя изменения должны вступать в силу при следующем входе пользователя в систему.
5. При ошибках валидации (неуникальный логин, некорректный формат данных и т.п.) система должна:
   - не применять изменения;
   - подсветить поля с ошибками;
   - отобразить сообщения об ошибке.

---

## REQ_USER_004 — Удаление пользователя (ADMIN)

1. Пользователь с ролью "ADMIN" должен иметь возможность удалить пользователя.
2. Перед удалением система должна запрашивать подтверждение действия (модальное окно с текстом подтверждения).
3. После удаления пользователь не должен иметь возможность авторизоваться в системе.
4. Удалённый пользователь не должен отображаться в списке пользователей.
5. Если при удалении пользователя произошла ошибка, система должна:
   - не изменять состояние пользователя;
   - отобразить сообщение об ошибке администратору.

---

## REQ_USER_005 — Ограничение прав доступа

1. Пользователь с ролью "USER" не должен иметь доступа к административному функционалу (просмотр пользователей, создание, редактирование, удаление пользователей, управление ролями и т.п.).
2. Попытка прямого обращения обычного пользователя к административным разделам (по URL или другим способам) должна приводить к отказу в доступе (страница ошибки/редирект).
3. Доступ к функционалу управления пользователями (просмотр, создание, редактирование, удаление) должен быть доступен только пользователям с ролью "ADMIN".
4. Доступ к просмотру списка пользователей может быть разрешён пользователям с ролью "SELLER" в соответствии с REQ_USER_001, при этом SELLER не должен иметь возможности изменять или удалять пользователей.

## REQ_USER_006 — Особые права администратора при создании и редактировании пользователей

1. При создании и редактировании пользователей через административный интерфейс (операции REQ_USER_002 и REQ_USER_003) значения полей должны:
   - соответствовать общим требованиям к данным пользователя (логин, пароль, email и т.п.),
   - за исключением случаев, когда явно допускается отступление для администратора.

2. Администратор может создавать и изменять пользователей с паролем, не соответствующим требованиям REQ_AUTH_002 — Требования к паролю (например, временный пользователь с упрощённым паролем).

3. Исключение, указанное в п.2:
   - не отменяет обязательность уникальности логина;
   - не отменяет обязательность указания роли пользователя;
   - не позволяет оставлять обязательные поля пустыми.

4. При создании или изменении пользователя с нарушением требований к паролю система должна:
   - всё равно принять и сохранить данные;
   - не блокировать операцию;
