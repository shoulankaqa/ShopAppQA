## API_AUTH_001 — Регистрация пользователя

1. Назначение
   - Метод предназначен для самостоятельной регистрации нового пользователя в системе.

2. URL и HTTP‑метод
   - HTTP‑метод: POST
   - URL: /register

3. Доступ и авторизация
   - Метод доступен без авторизации (публичный endpoint).
   - Не требует access‑токена.

4. Тело запроса

   Формат: `application/json`  

   Обязательные поля:
   - `username` (string) — логин пользователя, должен быть уникален в системе согласно требованиям REQ_AUTH_001 - Требования к логину.
   - `password` (string) — пароль пользователя, должен соответствовать требованиям REQ_AUTH_002 — Требования к паролю.
   (в отличие от администратора, обычный пользователь не может обходить эти требования)

   Пример запроса:

```http
POST /register HTTP/1.1
Host: api.example.com
Content-Type: application/json

{
  "username": "john",
  "password": "Pass1!"
}
```

5. Ответ при успехе

   - Код ответа: `201 Created`.
   - Формат: `application/json`.
   - Структура ответа соответствует Swagger‑схеме `User`:
     - `id` (string, uuid) — уникальный идентификатор пользователя.
     - `username` (string) — логин пользователя.
     - `role` (string) — роль пользователя, одно из значений: `admin`, `seller`, `user`.

   Бизнес‑правило:  
   - При регистрации через этот endpoint создаётся пользователь с ролью по умолчанию (`user`).

   Пример успешного ответа:

```http
HTTP/1.1 201 Created
Content-Type: application/json

{
  "id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
  "username": "john",
  "role": "user"
}
```

6. Ошибки

   - `400 Bad Request` — неверный формат данных или пользователь уже существует.
     - Причины:
       - не передано одно из обязательных полей (`username`, `password`);
       - данные не соответствуют требованиям формата (например, слишком короткий пароль, недопустимые символы, нарушены правила REQ_AUTH_002);
       - пользователь с таким `username` уже существует.
     - Тело ответа (application/json):
       - `code` (number) — код ошибки (400);
       - `message` (string) — описание ошибки (например, "User already exists" или "Invalid password format").

   Пример ответа при ошибке:

```http
HTTP/1.1 400 Bad Request
Content-Type: application/json

{
  "code": 400,
  "message": "User already exists"
}
```

   - `500 Internal Server Error` — внутренняя ошибка сервера (при наличии, формат ошибки может быть таким же: `code`, `message`).

7. Особые условия

   - При успешной регистрации пользователь должен иметь возможность авторизоваться, используя указанные `username` и `password` через endpoint авторизации (см. API_AUTH_002 — Авторизация пользователя).
   - Регистрация через этот endpoint не должна позволять пользователю самостоятельно выбрать повышенную роль (`admin`/`seller`).


## API_AUTH_002 — Авторизация пользователя (логин)

1. Назначение
   - Метод предназначен для авторизации пользователя по логину и паролю и выдачи пары токенов (access и refresh), а также информации о пользователе.

2. URL и HTTP‑метод
   - HTTP‑метод: POST
   - URL: /login

3. Доступ и авторизация
   - Метод доступен без предварительной авторизации (публичный endpoint).
   - Не требует access‑токена.

4. Тело запроса

   Формат: `application/json`

   Обязательные поля:
   - `username` (string) — логин пользователя.
   - `password` (string) — пароль пользователя.

   Требования:
   - Пара `username` + `password` должна совпадать с существующей учётной записью.
   - Для удалённых пользователей авторизация должна отклоняться.

   Пример запроса:

```http
POST /login HTTP/1.1
Host: api.example.com
Content-Type: application/json

{
  "username": "john",
  "password": "Pass1!"
}
```

5. Ответ при успехе

   - Код ответа: `200 OK`.
   - Формат: `application/json`.

   Структура ответа:

```json
{
  "access_token": "string",
  "refresh_token": "string",
  "user": {
    "id": "string, uuid",
    "username": "string",
    "email": "string",
    "role": "user"
  }
}
```

   Описание полей:
   - `access_token` (string) — токен доступа, используемый для авторизации запросов к защищённым endpoint’ам.
   - `refresh_token` (string) — токен для обновления `access_token`.
   - `user` — объект с данными пользователя:
     - `id` (string, uuid) — идентификатор пользователя.
     - `username` (string) — логин пользователя.
     - `email` (string) — email пользователя (может быть `null` при отсутствии).
     - `role` (string) — роль пользователя (`admin`, `seller`, `user`).

   Пример успешного ответа:

```http
HTTP/1.1 200 OK
Content-Type: application/json

{
  "access_token": "eyJhbGciOi...",
  "refresh_token": "eyJhbGciOi...",
  "user": {
    "id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
    "username": "john",
    "email": "john@example.com",
    "role": "user"
  }
}
```

6. Ошибки

   - `401 Unauthorized` — неверные учётные данные.
     - Причины:
       - `username` не существует;
       - пароль не совпадает с сохранённым для указанного `username`.
     - Тело ответа (application/json):

```json
{
  "code": 401,
  "message": "Invalid credentials"
}
```

   - `400 Bad Request` — (опционально, если реализовано) некорректный формат тела запроса, отсутствуют обязательные поля.
   - `500 Internal Server Error` — внутренняя ошибка сервера.

7. Особые условия

   - Возвращённый `access_token` должен подходить для использования в заголовке `Authorization: Bearer <access_token>` при обращении к защищённым API.
   - Возвращённый `refresh_token` должен использоваться для получения новых токенов через отдельный endpoint обновления (см. API_AUTH_003 — Обновление accsess токенов).
   - При нескольких успешных авторизациях подряд (например, с разных устройств) все выданные refresh‑токены считаются валидными до момента их истечения или явной инвалидции.


## API_AUTH_003 — Обновление access токена

1. Назначение
   - Метод предназначен для выдачи нового `access_token` по действующему `refresh_token` без повторного ввода логина и пароля.

2. URL и HTTP‑метод
   - HTTP‑метод: POST
   - URL: /refresh

3. Доступ и авторизация
   - Метод доступен без `Authorization: Bearer ...` заголовка.
   - Авторизация выполняется по переданному в теле запроса `refresh_token`.

4. Тело запроса

   Формат: `application/json`

   Обязательные поля:
   - `refresh_token` (string) — действующий refresh‑токен, полученный ранее при авторизации или обновлении токенов.

   Пример запроса:

```http
POST /refresh HTTP/1.1
Host: api.example.com
Content-Type: application/json

{
  "refresh_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
}
```

5. Ответ при успехе

   - Код ответа: `200 OK`.
   - Формат: `application/json`.
   - Структура ответа (согласно схеме `TokenResponse`):

```json
{
  "access_token": "string"
}
```

   Описание полей:
   - `access_token` (string) — новый токен доступа, который должен использоваться в заголовке `Authorization: Bearer <access_token>` для вызова защищённых методов.

   Пример успешного ответа:

```http
HTTP/1.1 200 OK
Content-Type: application/json

{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
}
```

6. Ошибки

   - `400 Bad Request` — неверные данные.
     - Причины:
       - отсутствует поле `refresh_token`;
       - некорректный формат тела запроса.
     - Тело ответа (application/json):

```json
{
  "code": 400,
  "message": "Invalid request data"
}
```

   - `401 Unauthorized` — не авторизован.
     - Причины:
       - `refresh_token` недействителен, истёк или отозван;
       - `refresh_token` не принадлежит ни одному пользователю.
     - Тело ответа (application/json):

```json
{
  "code": 401,
  "message": "Invalid or expired refresh token"
}
```

   - `500 Internal Server Error` — внутренняя ошибка сервера.

7. Особые условия

   - Новый `access_token` должен быть валиден для всех защищённых endpoint’ов так же, как токен, выданный методом `/login`.


## API_AUTH_004 — Выход пользователя (logout)

1. Назначение
   - Метод предназначен для выхода пользователя из системы за счёт инвалидции текущего сохранённого `refresh_token`, чтобы пользователь не мог далее обновлять `access_token` без повторной авторизации.

2. URL и HTTP‑метод
   - HTTP‑метод: POST
   - URL: /logout

3. Доступ и авторизация
   - Метод доступен только авторизованным пользователям.
   - Запрос должен выполняться с валидным `access_token` в заголовке `Authorization: Bearer <access_token>`.

4. Тело запроса
   - Тело запроса отсутствует.
   - Дополнительные параметры не требуются.

5. Ответ при успехе

   - Код ответа: `204 No Content`.
   - Тело ответа отсутствует.
   - Поведение:
     - сохранённый `refresh_token` текущего пользователя должен быть удалён/инвалидирован;
     - дальнейшие запросы на `/refresh` с этим `refresh_token` должны завершаться ошибкой `401 Unauthorized`;
     - текущий `access_token` может оставаться валидным до истечения срока жизни либо быть немедленно инвалидирован — выбранное поведение описывается в общих требованиях к безопасности (например, REQ_AUTH_004 — Политика токенов).

6. Ошибки

   - `401 Unauthorized` — отсутствует или некорректный `access_token`.
     - Тело ответа (application/json):

```json
{
  "code": 401,
  "message": "Invalid or missing access token"
}
```

7. Особые условия

   - Если у пользователя на момент вызова метода сохранённый `refresh_token` уже отсутствует (пользователь ранее разлогинен или токен уже инвалидирован), метод должен:
     - корректно завершиться с кодом `204 No Content`;
     - не считать это ошибкой (logout является идемпотентной операцией).
   - Метод влияет только на токены текущего пользователя и **не должен** затрагивать `refresh_token` других пользователей.
