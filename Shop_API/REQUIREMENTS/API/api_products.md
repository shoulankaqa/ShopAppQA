
## API_PRODUCTS_001 — Получение списка товаров с фильтрацией и пагинацией

1.1. Назначение: получение списка товаров с возможностью фильтрации по имени и доступности, а также с пагинацией.

1.2. Метод и URL:
- GET /products

1.3. Доступ:
- Метод доступен только авторизованным пользователям (`user`, `seller`, `admin`).
- Требуется валидный `access_token` в заголовке `Authorization: Bearer <access_token>`.

1.4. Query-параметры:
- `name` (string, optional) — фильтр по имени товара.
- `available` (boolean, optional) — фильтр по доступности товара.
- `limit` (integer, optional) — количество товаров на странице, по умолчанию 20.
- `offset` (integer, optional) — смещение, по умолчанию 0.

1.4.1. Параметр name:
- Опциональный.
- Тип: string.
- Если передан, возвращаются только товары, имя которых содержит указанное значение или соответствует правилам фильтрации, заданным в общих требованиях.

1.4.2. Параметр available:
- Опциональный.
- Тип: boolean.
- `true` — только доступные товары.
- `false` — только недоступные товары.

1.4.3. Параметр limit:
- Опциональный.
- Тип: integer.
- Минимум: 1.
- По умолчанию: 20.
- Максимум определяется общими требованиями к системе.

1.4.4. Параметр offset:
- Опциональный.
- Тип: integer.
- Минимум: 0.
- По умолчанию: 0.

1.5. Пример запроса:
```http
GET /products?name=Apple&available=true&limit=50&offset=0 HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token>
```

1.6. Успех:
- 200 OK.
- Формат ответа: application/json.
- В ответе возвращается список товаров с параметрами пагинации.

1.6.1. Пример успешного ответа:
```json
{
  "products": [
    {
      "id": 0,
      "name": "string",
      "price": 1,
      "available": true
    }
  ],
  "total": 0,
  "limit": 20,
  "offset": 0,
  "hasMore": false
}
```

1.6.2. Поля ответа:
- `products` — массив товаров.
- `products[].id` — идентификатор товара.
- `products[].name` — наименование товара.
- `products[].price` — цена товара.
- `products[].available` — признак доступности товара.
- `total` — общее количество товаров после фильтрации.
- `limit` — применённый лимит.
- `offset` — применённое смещение.
- `hasMore` — признак наличия следующих товаров.

1.7. Ошибки:
- 1.7.1. 401 Unauthorized — пользователь не авторизован или `access_token` отсутствует/некорректен.
- 1.7.2. 400 Bad Request — некорректные параметры запроса (`limit < 1`, `offset < 0`, неверный тип `available`).
- 1.7.3. 500 Internal Server Error — внутренняя ошибка сервера.

1.8. Особые условия:
- Сначала применяется фильтрация, затем пагинация.
- При пустом результате `products` должен быть пустым массивом, `total = 0`, `hasMore = false`.



## API_PRODUCTS_002 — Создание товара (SELLER, ADMIN)

2.1. Назначение: создание нового товара в системе.

2.2. Метод и URL:
- POST /products

2.3. Доступ:
- Метод доступен только авторизованным пользователям с ролями `seller` и `admin`.
- Требуется валидный `access_token` в заголовке `Authorization: Bearer <access_token>`.

2.4. Тело запроса:
- Формат: application/json.
- Схема: `ProductCreate`.
- Обязательные поля: `name`, `price`, `available`.

2.4.1. Поле name:
- Обязательно.
- Тип: string.
- Не должно быть пустым.
- Ограничения по длине и символам задаются в общих требованиях.

2.4.2. Поле price:
- Обязательно.
- Тип: number.
- Минимум: 1.

2.4.3. Поле available:
- Обязательно.
- Тип: boolean.
- `true` — товар доступен для покупки.
- `false` — товар недоступен.

2.5. Пример запроса:
```http
POST /products HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token>
Content-Type: application/json

{
  "name": "Яблоко красное",
  "price": 100,
  "available": true
}
```

2.6. Успех:
- 201 Created.
- Формат ответа: application/json.
- Ответ соответствует созданному товару.

2.6.1. Пример успешного ответа:
```json
{
  "id": 0,
  "name": "string",
  "price": 1,
  "available": true
}
```

2.7. Ошибки:
- 2.7.1. 400 Bad Request — отсутствует одно из обязательных полей, неверный формат или недопустимое значение.
- 2.7.2. 401 Unauthorized — пользователь не авторизован или `access_token` отсутствует/некорректен.
- 2.7.3. 403 Forbidden — у пользователя недостаточно прав для создания товара.
- 2.7.4. 500 Internal Server Error — внутренняя ошибка сервера.

2.8. Особые условия:
- Созданный товар должен попадать в выборку `GET /products` с учётом фильтрации и пагинации.



## API_PRODUCTS_003 — Получение товара по ID

3.1. Назначение: получение подробной информации о товаре по его идентификатору.

3.2. Метод и URL:
- GET /products/{id}

3.3. Доступ:
- Метод доступен только авторизованным пользователям (`user`, `seller`, `admin`).
- Требуется валидный `access_token` в заголовке `Authorization: Bearer <access_token>`.

3.4. Параметры запроса:
- Path-параметр `id` (integer, required) — идентификатор товара.

3.5. Пример запроса:
```http
GET /products/42 HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token>
```

3.6. Успех:
- 200 OK.
- Формат ответа: application/json.

3.6.1. Пример успешного ответа:
```json
{
  "id": 0,
  "name": "string",
  "price": 0,
  "available": true
}
```

3.6.2. Поля ответа:
- `id` — идентификатор товара.
- `name` — наименование товара.
- `price` — цена товара.
- `available` — признак доступности товара.

3.7. Ошибки:
- 3.7.1. 400 Bad Request — некорректный `id`.
- 3.7.2. 401 Unauthorized — пользователь не авторизован или `access_token` отсутствует/некорректен.
- 3.7.3. 404 Not Found — товар не найден.
- 3.7.4. 500 Internal Server Error — внутренняя ошибка сервера.

3.8. Особые условия:
- Метод возвращает товар независимо от его доступности.



## API_PRODUCTS_004 — Полное обновление товара (SELLER, ADMIN)

4.2. Метод и URL:

- Path-параметр `id` (integer, required) — идентификатор товара.

4.5. Тело запроса:
- Формат: application/json.
- Схема: `ProductCreate`.
- Обязательные поля: `name`, `price`, `available`.

4.5.1. Поле name:
- Обязательно.
- Тип: string.
- Не должно быть пустым.

4.5.2. Поле price:
- Обязательно.
- Тип: number.
- Минимум: 1.

4.5.3. Поле available:
- Обязательно.
- Тип: boolean.

4.6. Пример запроса:
```http
PUT /products/42 HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token>
Content-Type: application/json

{
  "name": "Яблоко красное большое",
  "price": 150,
  "available": true
}
```

- 200 OK.
- Формат ответа: application/json.
- Возвращается обновлённый товар.

4.7.1. Пример успешного ответа:
```json
{

  "name": "string",
  "price": 1,
  "available": true
}
```

- 4.8.1. 400 Bad Request — неверные данные или отсутствует обязательное поле.
- 4.8.2. 401 Unauthorized — пользователь не авторизован или `access_token` отсутствует/некорректен.
- 4.8.3. 403 Forbidden — у пользователя недостаточно прав для обновления товара.
- 4.8.4. 404 Not Found — товар не найден.
- 4.8.5. 500 Internal Server Error — внутренняя ошибка сервера.

4.9. Особые условия:
- PUT — полное обновление товара.
- Все поля товара должны быть заменены значениями из запроса.



## API_PRODUCTS_005 — Частичное обновление товара (SELLER, ADMIN)

5.1. Назначение: частичное обновление данных товара по идентификатору.

5.2. Метод и URL:
- PATCH /products/{id}

5.3. Доступ:
- Метод доступен только авторизованным пользователям с ролями `seller` и `admin`.
- Требуется валидный `access_token` в заголовке `Authorization: Bearer <access_token>`.

5.4. Параметры запроса:
- Path-параметр `id` (integer, required) — идентификатор товара.

5.5. Тело запроса:
- Формат: application/json.
- Схема: `ProductUpdate`.
- Поля: `name`, `price`, `available` — все optional.

5.5.1. Поле name:
- Опциональное.
- Тип: string.
- Не должно быть пустым, если передано.

5.5.2. Поле price:
- Опциональное.
- Тип: number.
- Минимум: 1, если передано.

5.5.3. Поле available:
- Опциональное.
- Тип: boolean.

5.6. Пример запроса:
```http
PATCH /products/42 HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token>
Content-Type: application/json

{
  "price": 180,
  "available": false
}
```

5.7. Успех:
- 200 OK.
- Формат ответа: application/json.
- Возвращается обновлённый товар.

5.7.1. Пример успешного ответа:
```json
{
  "id": 0,
  "name": "string",
  "price": 1,
  "available": true
}
```

5.8. Ошибки:
- 5.8.1. 400 Bad Request — неверные данные или не передано ни одно поле.
- 5.8.2. 401 Unauthorized — пользователь не авторизован или `access_token` отсутствует/некорректен.
- 5.8.3. 403 Forbidden — у пользователя недостаточно прав для обновления товара.
- 5.8.4. 404 Not Found — товар не найден.
- 5.8.5. 500 Internal Server Error — внутренняя ошибка сервера.

5.9. Особые условия:
- PATCH — частичное обновление товара.
- Изменяются только переданные поля.



## API_PRODUCTS_006 — Удаление товара (только ADMIN)

6.1. Назначение: полное удаление товара по идентификатору.

6.2. Метод и URL:
- DELETE /products/{id}

6.3. Доступ:
- Метод доступен только авторизованным пользователям с ролью `admin`.
- Требуется валидный `access_token` в заголовке `Authorization: Bearer <access_token>`.

6.4. Параметры запроса:
- Path-параметр `id` (integer, required) — идентификатор товара.

6.5. Пример запроса:
```http
DELETE /products/42 HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token>
```

6.6. Успех:
- 204 No Content.
- Тело ответа отсутствует.

6.7. Ошибки:
- 6.7.1. 400 Bad Request — некорректный `id`.
- 6.7.2. 401 Unauthorized — пользователь не авторизован или `access_token` отсутствует/некорректен.
- 6.7.3. 403 Forbidden — у пользователя недостаточно прав для удаления товара.
- 6.7.4. 404 Not Found — товар не найден.
- 6.7.5. 500 Internal Server Error — внутренняя ошибка сервера.

6.8. Особые условия:
- Удаление товара необратимо.
- Повторный DELETE по уже удалённому `id` должен возвращать `404 Not Found`.
