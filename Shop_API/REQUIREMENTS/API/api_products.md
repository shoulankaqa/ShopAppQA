## API_PRODUCTS_001 — Получение списка товаров с фильтрацией и пагинацией

1. Назначение
   - Метод предназначен для получения списка всех товаров с возможностью фильтрации по имени и доступности, а также с пагинацией (например, для отображения в каталоге).

2. URL и HTTP‑метод
   - HTTP‑метод: GET
   - URL: /products

3. Доступ и авторизация
   - Метод доступен только авторизованным пользователям (всем ролям, которые должны видеть каталог: `user`, `seller`, `admin`).
   - Запрос должен выполняться с валидным `access_token` в заголовке `Authorization: Bearer <access_token>`.

4. Query‑параметры

   - `name` (string, optional) — фильтр по имени товара.
     - Если указан, возвращается только товары, имя которых содержит переданное значение (или строго совпадает — точное поведение фильтрации определяется в общих требованиях к продуктам).
   - `available` (boolean, optional) — фильтр по доступности товара.
     - `true` — только доступные товары (`available = true`);
     - `false` — только недоступные товары (`available = false`).
   - `limit` (integer, optional) — количество товаров на странице.
     - По умолчанию: `20`.
     - Минимальное значение: 1.
     - Максимальное значение: определяется в общих требованиях (например, до 100).
   - `offset` (integer, optional) — номер сдвига (начало страницы).
     - По умолчанию: `0`.
     - Минимальное значение: 0.

   Пример запроса:

```http
GET /products?name=Apple&available=true&limit=50&offset=0 HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token>
```

5. Ответ при успехе

   - Код ответа: `200 OK`.
   - Формат: `application/json`.

   Структура ответа:

```json
{
  "products": [
    {
      "id": 0,
      "name": "string",
      "price": 0,
      "available": true
    }
  ],
  "total": 0,
  "limit": 100,
  "offset": 0,
  "hasMore": true
}
```

   Описание полей:
   - `products` (array) — список товаров.
     - `products[].id` (integer) — идентификатор товара.
     - `products[].name` (string) — наименование товара.
     - `products[].price` (number) — цена товара.
     - `products[].available` (boolean) — признак доступности товара для покупки.
   - `total` (integer) — общее количество товаров, соответствующих фильтру.
   - `limit` (integer) — использованный лимит товаров на странице.
   - `offset` (integer) — использованный сдвиг.
   - `hasMore` (boolean) — флаг, указывающий, есть ли ещё товары после текущей страницы.

6. Ошибки

   - `401 Unauthorized` — пользователь не авторизован или `access_token` отсутствует/некорректен.

```json
{
  "code": 401,
  "message": "string"
}
```

   - (Опционально) `400 Bad Request` — некорректные параметры запроса (например, `limit < 1`, `offset < 0`, некорректный тип `available`).
   - (Опционально) `500 Internal Server Error` — внутренняя ошибка сервера.

7. Особые условия

   - Фильтрация и пагинация должны работать совместно:
     - сначала применяется фильтр (`name`, `available`),
     - затем отфильтрованный результат обрезается по `limit` и `offset`.
   - `total` — это количество элементов **после** фильтрации, но **перед** обрезкой по `limit`/`offset`.
   - Поведение при пустом результате:
     - возвращается `products` с пустым массивом,
     - `total = 0`,
     - `hasMore = false`.

    ## API_PRODUCTS_002 — Создание товара (SELLER, ADMIN)

1. Назначение
   - Метод предназначен для создания нового товара в системе пользователями с правами продавца или администратора.

2. URL и HTTP‑метод
   - HTTP‑метод: POST
   - URL: /products

3. Доступ и авторизация
   - Метод доступен только авторизованным пользователям с ролями `seller` и `admin`.
   - Запрос должен выполняться с валидным `access_token` в заголовке `Authorization: Bearer <access_token>`.

4. Тело запроса

   Формат: `application/json`  
   Схема: `ProductCreate`.

   Обязательные поля:
   - `name` (string) — наименование товара.
   - `price` (number, minimum: 0) — цена товара.
   - `available` (boolean) — признак доступности товара для покупки.

   Правила:
   - `name` не должен быть пустым (конкретные ограничения по длине и допустимым символам задаются в функциональных требованиях к товарам).
   - `price` должен быть числом и `>= 0`:
     - если по бизнес‑логике цена не может быть 0, это нужно дополнительно зафиксировать (например, `price > 0`).
   - `available = true` означает, что товар доступен для добавления в корзину; `false` — недоступен.

   Пример запроса:

```http
POST /products HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token>
Content-Type: application/json

{
  "name": "Яблоко красное",
  "price": 100,
  "available": true
}
```

5. Ответ при успехе

   - Код ответа: `201 Created`.
   - Формат: `application/json`.

```json
{
  "id": 0,
  "name": "string",
  "price": 0,
  "available": true
}
```

   Описание полей:
   - `id` (integer) — идентификатор созданного товара.
   - `name` (string) — наименование товара.
   - `price` (number) — цена товара.
   - `available` (boolean) — актуальный признак доступности товара.

6. Ошибки

   - `400 Bad Request` — неверные данные.

```json
{
  "code": 400,
  "message": "string"
}
```

   Возможные причины:
   - отсутствует одно из обязательных полей (`name`, `price`, `available`);
   - `price` имеет недопустимое значение (например, меньше 0 или не число);
   - `name` нарушает ограничения по формату/длине (если заданы).

   - `401 Unauthorized` — пользователь не авторизован или `access_token` недействителен/отсутствует (общие правила).
   - `403 Forbidden` — пользователь авторизован, но его роль не `seller` и не `admin` (если такая проверка реализована).
   - `500 Internal Server Error` — внутренняя ошибка сервера.

7. Особые условия

   - Созданный товар должен появляться в выборке `GET /products` в соответствии с параметрами фильтрации (например, по `available`).
   - Правила валидации полей товара (ограничения по имени, ценам и т.п.) должны быть согласованы с функциональными требованиями к модулю товаров (REQ_PRODUCT_002).


## API_PRODUCTS_003 — Получение товара по ID

1. Назначение
   - Метод предназначен для получения подробной информации об одном товаре по его идентификатору.

2. URL и HTTP‑метод
   - HTTP‑метод: GET
   - URL: /products/{id}

3. Доступ и авторизация
   - Метод доступен только авторизованным пользователям (всем ролям, которые должны иметь доступ к каталогам: `user`, `seller`, `admin`).
   - Запрос должен выполняться с валидным `access_token` в заголовке `Authorization: Bearer <access_token>`.

4. Параметры запроса

   Path‑параметры:
   - `id` (integer, required) — идентификатор товара, который должен быть получен.

   Пример запроса:

```http
GET /products/42 HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token>
```

5. Ответ при успехе

   - Код ответа: `200 OK`.
   - Формат: `application/json`.

```json
{
  "id": 0,
  "name": "string",
  "price": 0,
  "available": true
}
```

   Описание полей:
   - `id` (integer) — идентификатор товара.
   - `name` (string) — наименование товара.
   - `price` (number) — цена товара.
   - `available` (boolean) — признак доступности товара для покупки.

6. Ошибки

   - `400 Bad Request` — некорректный ID (не integer, недопустимый формат).

```json
{
  "code": 400,
  "message": "Invalid ID"
}
```

   - `401 Unauthorized` — пользователь не авторизован или `access_token` отсутствует/некорректен.

```json
{
  "code": 401,
  "message": "string"
}
```

   - `404 Not Found` — товар не найден (заданный `id` не принадлежит ни одному товару).

```json
{
  "code": 404,
  "message": "Product not found"
}
```

   - (Опционально) `500 Internal Server Error` — внутренняя ошибка сервера.

7. Особые условия

   - Метод возвращает товар независимо от его доступности (`available`), но поле `available` должно быть в ответе, чтобы клиент мог корректно отображать статус товара.


## API_PRODUCTS_004 — Полное обновление товара (SELLER, ADMIN)

1. Назначение
   - Метод предназначен для полного обновления данных товара по идентификатору: все доступные поля товара заменяются значениями из запроса.

2. URL и HTTP‑метод
   - HTTP‑метод: PUT
   - URL: /products/{id}

3. Доступ и авторизация
   - Метод доступен только авторизованным пользователям с ролями `seller` и `admin`.
   - Запрос должен выполняться с валидным `access_token` в заголовке `Authorization: Bearer <access_token>`.

4. Параметры запроса

   Path‑параметры:
   - `id` (integer, required) — идентификатор товара, который нужно обновить.

   Пример запроса:

```http
PUT /products/42 HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token>
Content-Type: application/json

{
  "name": "Яблоко красное большое",
  "price": 150,
  "available": true
}
```

5. Тело запроса

   Формат: `application/json`  
   Схема: `ProductCreate` (аналогично созданию товара).

   Обязательные поля:
   - `name` (string) — новое наименование товара.
   - `price` (number, minimum: 0) — новая цена товара.
   - `available` (boolean) — новый признак доступности товара.

   Правила:
   - Все поля обязательны; при отсутствии любого из них возвращается `400 Bad Request`.
   - `name` не должен быть пустым (ограничения по длине и символам — в функциональных требованиях).
   - `price` должно быть числом и `>= 0` (если бизнес‑логика запрещает 0, это зафиксировать отдельно).
   - `available = true` — товар доступен для покупки; `false` — недоступен.

6. Ответ при успехе

   - Код ответа: `200 OK`.
   - Формат: `application/json`.

```json
{
  "id": 0,
  "name": "string",
  "price": 0,
  "available": true
}
```

   Описание полей:
   - `id` (integer) — идентификатор товара (не меняется).
   - `name` (string) — актуальное наименование товара после обновления.
   - `price` (number) — актуальная цена товара.
   - `available` (boolean) — актуальный признак доступности товара.

7. Ошибки

   - `400 Bad Request` — неверные данные.

```json
{
  "code": 400,
  "message": "string"
}
```

   Возможные причины:
   - отсутствует одно из обязательных полей;
   - `price` недопустимое значение (меньше 0 или не число);
   - `name` не соответствует ограничениям по формату/длине.

   - `401 Unauthorized` — пользователь не авторизован или `access_token` недействителен/отсутствует.
   - `403 Forbidden` — пользователь авторизован, но его роль не `seller` и не `admin` (если такая проверка реализована).
   - `404 Not Found` — товар не найден (заданный `id` не принадлежит ни одному товару).

```json
{
  "code": 404,
  "message": "string"
}
```

   - (Опционально) `500 Internal Server Error` — внутренняя ошибка сервера.

8. Особые условия

   - PUT — полное обновление: все поля товара заменяются, незаполненные поля не сохраняются (в отличие от PATCH, если он будет реализован).
   - Обновлённый товар должен появляться в выборке `GET /products` в соответствии с параметрами фильтрации и пагинации.
   - Правила валидации полей товара должны быть согласованы с функциональными требованиями к модулю товаров (REQ_PRODUCT_002).


## API_PRODUCTS_005 — Частичное обновление товара (SELLER, ADMIN)

1. Назначение
   - Метод предназначен для частичного обновления данных товара по идентификатору: изменяются только переданные поля, остальные остаются без изменений.

2. URL и HTTP‑метод
   - HTTP‑метод: PATCH
   - URL: /products/{id}

3. Доступ и авторизация
   - Метод доступен только авторизованным пользователям с ролями `seller` и `admin`.
   - Запрос должен выполняться с валидным `access_token` в заголовке `Authorization: Bearer <access_token>`.

4. Параметры запроса

   Path‑параметры:
   - `id` (integer, required) — идентификатор товара, который нужно обновить.

   Пример запроса:

```http
PATCH /products/42 HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token>
Content-Type: application/json

{
  "price": 180,
  "available": false
}
```

5. Тело запроса

   Формат: `application/json`  
   Схема: `ProductUpdate`.

   Поля (все optional):
   - `name` (string) — новое (или частичное) наименование товара.
   - `price` (number, minimum: 0) — новая цена товара.
   - `available` (boolean) — новый признак доступности товара.

   Правила:
   - одно поле должно быть передано; если тело пустое или все поля не переданы, может возвращаться `400 Bad Request`.
   - `name` не должен быть пустым (если передан).
   - `price` должно быть числом и `>= 0` (если передан).
   - `available` должно быть `true` или `false` (если передано).

6. Ответ при успехе

   - Код ответа: `200 OK`.
   - Формат: `application/json`.

```json
{
  "id": 0,
  "name": "string",
  "price": 0,
  "available": true
}
```

   Описание полей:
   - `id` (integer) — идентификатор товара.
   - `name` (string) — актуальное наименование товара после обновления.
   - `price` (number) — актуальная цена товара.
   - `available` (boolean) — актуальный признак доступности товара.

7. Ошибки

   - `400 Bad Request` — неверные данные.

```json
{
  "code": 400,
  "message": "string"
}
```

   Возможные причины:
   - переданы недопустимые значения полей;
   - `price` < 0 или не число;
   - `name` пустой (если передан).

   - `401 Unauthorized` — пользователь не авторизован или `access_token` недействителен/отсутствует.
   - `403 Forbidden` — пользователь авторизован, но его роль не `seller` и не `admin`.
   - `404 Not Found` — товар не найден (заданный `id` не принадлежит ни одному товару).

```json
{
  "code": 404,
  "message": "string"
}
```

8. Особые условия

   - PATCH — частичное обновление: только переданные поля заменяются, остальные сохраняются.
   - Обновлённый товар должен появляться в выборке `GET /products` с учётом фильтрации и пагинации.
   - Правила валидации должны быть согласованы с функциональными требованиями к модулю товаров (REQ_PRODUCT_003).


## API_PRODUCTS_006 — Удаление товара (только ADMIN)

1. Назначение
   - Метод предназначен для полного удаления товара по идентификатору.
   - Удаление доступно только пользователю с ролью `admin`.


2. URL и HTTP‑метод
   - HTTP‑метод: DELETE
   - URL: /products/{id}


3. Доступ и авторизация
   - Метод доступен только авторизованным пользователям с ролью `admin`.
   - Запрос должен выполняться с валидным `access_token` в заголовке `Authorization: Bearer <access_token>`.


4. Параметры запроса


   Path‑параметры:
   - `id` (integer, required) — идентификатор товара, который нужно удалить.


   Пример запроса:

```http
DELETE /products/42 HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token>
```

5. Тело запроса
   - Тело запроса не требуется (метод `DELETE` не передаёт данные).


6. Ответ при успехе

   - Код ответа: `204 No Content`.
   - Формат: без тела ответа (No Content).

   Описание:
   - Товар успешно удалён из системы.


7. Ошибки

   - `401 Unauthorized` — пользователь не авторизован или `access_token` недействителен/отсутствует.

```json
{
  "code": 401,
  "message": "string"
}
```

   Возможные причины:
   - отсутствует заголовок `Authorization`;
   - `access_token` недействителен или истёк.

   - `403 Forbidden` — пользователь авторизован, но его роль не `admin`.


```json
{
  "code": 403,
  "message": "string"
}
```

   - `404 Not Found` — товар не найден (заданный `id` не принадлежит ни одному товару).

```json
{
  "code": 404,
  "message": "string"
}
```

   Возможные причины:
   - товар с таким `id` уже удалён;
   - `id` не соответствует ни одному существующему товару.
   - `400 Bad Request` (опционально) — неверный формат `id` (например, не число).

```json
{
  "code": 400,
  "message": "string"
}
```

8. Особые условия

   - DELETE — полное удаление: товар удаляется из таблицы товаров без возможности восстановления.
   - Повторный `DELETE` по уже удалённому `id` должен возвращать `404 Not Found` (идемпотентность операции).
   - Правила валидации и поведения при ошибках должны быть согласованы с функциональными требованиями к модулю товаров (REQ_PRODUCT_004)
