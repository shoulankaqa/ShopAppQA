## API_CART_001 — Получение корзины текущего пользователя

1.1. Назначение: получение содержимого корзины текущего авторизованного пользователя.

1.2. Метод и URL:
- GET /cart

1.3. Доступ:
- Метод доступен только авторизованным пользователям (ролям, которым разрешена работа с корзиной — `user`, `seller`, `admin`).
- Требуется валидный `access_token` в заголовке `Authorization: Bearer <access_token>`.

1.4. Тело запроса:
- Отсутствует.
- Дополнительные параметры не используются.

1.5. Успех:
- 200 OK.
- Формат ответа: application/json.
- В ответе возвращается объект корзины текущего пользователя.

1.5.1. Структура ответа (пример):
```json
{
  "id": 0,
  "items": [
    {
      "id": 0,
      "product": {
        "id": 0,
        "name": "string",
        "price": 1,
        "available": true
      },
      "quantity": 0
    }
  ]
}
```

1.5.2. Поля ответа:
- `id` — идентификатор корзины текущего пользователя.
- `items` — массив позиций корзины.
- `items[].id` — идентификатор позиции корзины.
- `items[].product` — объект товара (id, name, price, available).
- `items[].quantity` — количество единиц товара в позиции.

1.6. Правила:
- Если в корзине нет ни одной позиции, возвращается объект корзины с пустым массивом `items` (а не ошибка).

1.7. Ошибки:
- 1.7.1. 401 Unauthorized — пользователь не авторизован или `access_token` отсутствует/некорректен.
- 1.7.2. 403 Forbidden — у текущей роли недостаточно прав для работы с корзиной.
- 1.7.3. 500 Internal Server Error — внутренняя ошибка сервера.

1.8. Особые условия:
- Метод возвращает корзину именно текущего пользователя, определённого по `access_token`; передавать идентификаторы пользователя или корзины в запросе нельзя.



## API_CART_002 — Добавление товара в корзину или изменение его количества

2.1. Назначение: добавление товара в корзину текущего пользователя или изменение количества уже добавленного товара.

2.2. Метод и URL:
- POST /cart/items

2.3. Доступ:
- Метод доступен только авторизованным пользователям (`user`, `seller`, `admin`).
- Требуется валидный `access_token` в заголовке `Authorization: Bearer <access_token>`.

2.4. Тело запроса:
- Формат: application/json.
- Обязательные поля: `productId` (integer), `quantity` (integer).

2.4.1. Правила:
- `productId` и `quantity` обязательны.
- При `quantity > 0`:
  - если товара ещё нет в корзине — он добавляется с указанным количеством;
  - если товар уже есть в корзине — его количество изменяется согласно бизнес‑логике (перезапись/увеличение/уменьшение — уточняется в общих требованиях).
- При `quantity <= 0` поведение определяется отдельными бизнес‑правилами (ошибка 400 или удаление позиции).
- Товар может быть добавлен/изменён только если он существует и доступен для покупки (например, `available = true`).

2.5. Пример запроса:
```http
POST /cart/items HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token>
Content-Type: application/json

{
  "productId": 42,
  "quantity": 3
}
```

2.6. Успех:
- 200 OK.
- Формат ответа: application/json.
- Возвращается объект позиции корзины после выполнения операции.

2.6.1. Пример успешного ответа:
```json
{
  "id": 0,
  "product": {
    "id": 0,
    "name": "string",
    "price": 0,
    "available": true
  },
  "quantity": 0
}
```

2.7. Ошибки:
- 2.7.1. 400 Bad Request — некорректный запрос (обязательные поля не переданы, неверные значения, `quantity` вне допустимого диапазона).
- 2.7.2. 401 Unauthorized — пользователь не авторизован или `access_token` отсутствует/некорректен.
- 2.7.3. 403 Forbidden — у текущей роли недостаточно прав для работы с корзиной.
- 2.7.4. 404 Not Found — товар не найден или недоступен для покупки.
- 2.7.5. 500 Internal Server Error — внутренняя ошибка сервера.

2.8. Особые условия:
- Метод работает только с корзиной текущего пользователя, определённого по `access_token`.



## API_CART_003 — Изменение количества товара в корзине

3.1. Назначение: изменение количества товара в корзине текущего пользователя по идентификатору позиции корзины.

3.2. Метод и URL:
- PATCH /cart/items/{id}

3.3. Доступ:
- Метод доступен только авторизованным пользователям (`user`, `seller`, `admin`).
- Требуется валидный `access_token` в заголовке `Authorization: Bearer <access_token>`.

3.4. Параметры запроса:
- Path‑параметр `id` (uuid) — идентификатор позиции корзины, которую нужно изменить или удалить.

3.5. Тело запроса:
- Формат: application/json.
- Обязательное поле: `quantity` (integer, минимум 0).

3.6. Правила:
- `quantity` обязателен и должен быть `>= 0`.
- Если `quantity > 0`:
  - позиция корзины обновляется с указанным количеством;
  - возвращается обновлённая позиция (код 200 OK).
- Если `quantity == 0`:
  - позиция корзины удаляется;
  - возвращается 204 No Content, тело отсутствует.
- Значение `quantity < 0` считается некорректным и приводит к 400 Bad Request.

3.7. Пример запроса (изменение количества):
```http
PATCH /cart/items/04167fbf-c7a7-433a-997e-6394ae51ebb5 HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token>
Content-Type: application/json

{
  "quantity": 3
}
```

3.8. Пример запроса (удаление позиции):
```http
PATCH /cart/items/04167fbf-c7a7-433a-997e-6394ae51ebb5 HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token>
Content-Type: application/json

{
  "quantity": 0
}
```

3.9. Успех:
- При `quantity > 0`:
  - 200 OK.
  - Возвращается обновлённая позиция корзины.
- При `quantity == 0`:
  - 204 No Content.
  - Тело ответа отсутствует.

3.9.1. Пример успешного ответа (quantity > 0):
```json
{
  "id": 0,
  "product": {
    "id": 0,
    "name": "string",
    "price": 0,
    "available": true
  },
  "quantity": 0
}
```

3.10. Ошибки:
- 3.10.1. 400 Bad Request — некорректный запрос (отсутствует `quantity`, `quantity < 0`, некорректный формат тела).
- 3.10.2. 401 Unauthorized — пользователь не авторизован или `access_token` отсутствует/некорректен.
- 3.10.3. 403 Forbidden — у текущей роли недостаточно прав для работы с корзиной.
- 3.10.4. 404 Not Found — позиция корзины с указанным `id` не найдена или принадлежит другому пользователю.
- 3.10.5. 500 Internal Server Error — внутренняя ошибка сервера.

3.11. Особые условия:
- Метод работает только с позициями корзины текущего пользователя, определённого по `access_token`; операции с чужими корзинами должны приводить к 404 Not Found.



## API_CART_004 — Удаление позиции из корзины

4.1. Назначение: удаление указанной позиции из корзины текущего авторизованного пользователя.

4.2. Метод и URL:
- DELETE /cart/items/{id}

4.3. Доступ:
- Метод доступен только авторизованным пользователям (`user`, `seller`, `admin`).
- Требуется валидный `access_token` в заголовке `Authorization: Bearer <access_token>`.

4.4. Параметры запроса:
- Path‑параметр `id` (uuid) — идентификатор позиции корзины, которая должна быть удалена.

4.5. Пример запроса:
```http
DELETE /cart/items/04167fbf-c7a7-433a-997e-6394ae51ebb5 HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token>
```

4.6. Успех:
- 204 No Content.
- Тело ответа отсутствует.
- Позиция корзины с указанным `id` должна быть удалена и не возвращаться в ответе `GET /cart`.

4.7. Ошибки:
- 4.7.1. 400 Bad Request — неверные данные (например, некорректный формат `id`).
- 4.7.2. 401 Unauthorized — пользователь не авторизован или `access_token` отсутствует/некорректен.
- 4.7.3. 403 Forbidden — у текущей роли недостаточно прав для работы с корзиной.
- 4.7.4. 404 Not Found — позиция корзины не найдена или не принадлежит текущему пользователю.
- 4.7.5. 500 Internal Server Error — внутренняя ошибка сервера.

4.8. Особые условия:
- Метод удаляет только позиции корзины текущего пользователя, определённого по `access_token`; при попытке удалить позицию чужой корзины должна возвращаться 404 Not Found.
- Операция должна быть идемпотентной: повторный DELETE той же позиции после успешного удаления может возвращать 404 Not Found.



## API_CART_005 — Оплата корзины из кошелька

5.1. Назначение: оплата содержимого корзины текущего авторизованного пользователя за счёт средств его кошелька.

5.2. Метод и URL:
- POST /v2/cart/checkout

5.3. Доступ:
- Метод доступен только авторизованным пользователям (`user`, `seller`, `admin`), которым разрешена работа с корзиной и кошельком.
- Требуется валидный `access_token` в заголовке `Authorization: Bearer <access_token>`.

5.4. Тело запроса:
- Отсутствует.
- Все данные берутся из корзины и кошелька текущего пользователя.

5.5. Логика работы:
- Определить текущего пользователя по `access_token`.
- Получить корзину пользователя и рассчитать итоговую сумму к оплате.
- Проверить, что корзина не пуста.
- Получить текущий баланс кошелька пользователя.
- Если средств недостаточно — вернуть ошибку 409 Conflict.
- Если средств достаточно:
  - списать сумму корзины с баланса кошелька;
  - очистить корзину (удалить все позиции).

5.6. Успех:
- 200 OK.
- Формат ответа: application/json.
- Возвращается информация о результатах операции и новом балансе.

5.6.1. Пример успешного ответа:
```json
{
  "code": 200,
  "message": "Вы оплатили корзину на 1500 рублей",
  "newBalance": 8500
}
```

5.7. Ошибки:
- 5.7.1. 400 Bad Request — корзина пуста.
- 5.7.2. 401 Unauthorized — пользователь не авторизован или `access_token` отсутствует/некорректен.
- 5.7.3. 403 Forbidden — недостаточно прав для выполнения операции.
- 5.7.4. 409 Conflict — недостаточно средств на балансе кошелька; в ответе могут возвращаться `currentBalance` и `requiredAmount`.
- 5.7.5. 500 Internal Server Error — внутренняя ошибка сервера.

5.8. Особые условия:
- Метод работает только с корзиной и кошельком текущего пользователя.
- Операция должна быть атомарной (списание средств и очистка корзины выполняются как единая транзакция).
- Повторный вызов после успешной оплаты (при пустой корзине) должен возвращать 400 Bad Request `"Корзина пуста"`.



## API_CART_006 — Очистка корзины текущего пользователя

6.1. Назначение: полная очистка корзины текущего авторизованного пользователя (удаление всех позиций).

6.2. Метод и URL:
- DELETE /cart/clear

6.3. Доступ:
- Метод доступен только авторизованным пользователям (`user`, `seller`, `admin`).
- Требуется валидный `access_token` в заголовке `Authorization: Bearer <access_token>`.

6.4. Тело запроса:
- Отсутствует.
- Дополнительные параметры не используются.

6.5. Логика работы:
- Определить текущего пользователя по `access_token`.
- Получить его корзину.
- Если корзина пуста — вернуть 400 Bad Request `"Корзина пуста"`.
- Если корзина не пуста — удалить все позиции из корзины и вернуть успешный ответ.

6.6. Успех:
- 200 OK.
- Формат ответа: application/json.
- Пример успешного ответа:
```json
{
  "code": 200,
  "message": "Корзина очищена успешно"
}
```

6.7. Ошибки:
- 6.7.1. 400 Bad Request — корзина пуста.
- 6.7.2. 401 Unauthorized — пользователь не авторизован или `access_token` отсутствует/некорректен.
- 6.7.3. 403 Forbidden — недостаточно прав для выполнения операции.
- 6.7.4. 500 Internal Server Error — внутренняя ошибка сервера.

6.8. Особые условия:
- Метод работает только с корзиной текущего пользователя, определённого по `access_token`.
- Операция должна быть идемпотентной: при повторном вызове после успешной очистки снова возвращается 400 Bad Request `"Корзина пуста"`.
