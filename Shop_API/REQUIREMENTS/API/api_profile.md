## API_PROFILE_001 — Получение данных текущего пользователя (самостоятельно)

1. Назначение
   - Метод предназначен для получения данных профиля текущего авторизованного пользователя по его `access_token`.
   - Пользователь может через этот метод самостоятельно посмотреть на себя информацию о своём профиле (id, username, email, role).

2. URL и HTTP‑метод
   - HTTP‑метод: GET
   - URL: /me

3. Доступ и авторизация
   - Метод доступен только авторизованным пользователям (всем ролям: admin, seller, user).
   - Запрос должен выполняться с валидным `access_token` в заголовке `Authorization: Bearer <access_token>`.

4. Тело запроса
   - Отсутствует.

5. Ответ при успехе

   - Код ответа: `200 OK`.
   - Формат: `application/json`.

   Структура ответа:

```json
{
  "id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
  "username": "string",
  "email": "string",
  "role": "admin"
}
```

   Описание полей:
   - `id` (string, uuid) — идентификатор пользователя.
   - `username` (string) — логин пользователя.
   - `email` (string или null) — email пользователя.
   - `role` (string) — роль пользователя (`admin`, `seller`, `user`).

6. Ошибки

   - `401 Unauthorized` — пользователь не авторизован или `access_token` отсутствует/некорректен.
     - Тело ответа (пример):

```json
{
  "code": 401,
  "message": "Missing token"
}
```

7. Особые условия

   - Метод всегда возвращает данные только текущего пользователя, определённого по `access_token`; передавать `id` пользователя в запросе нельзя.

## API_PROFILE_002 — Обновление профиля текущего пользователя (самостоятельно)

1. Назначение
   - Метод предназначен для частичного обновления данных профиля текущего авторизованного пользователя email, имя и/или другие поля профиля, доступные для самостоятельного редактирования пользователем.

2. URL и HTTP‑метод
   - HTTP‑метод: PATCH
   - URL: /me

3. Доступ и авторизация
   - Метод доступен только авторизованным пользователям (всем ролям, которым разрешено самостоятельно обновлять профиль: admin, seller, user).
   - Запрос должен выполняться с валидным `access_token` в заголовке `Authorization: Bearer <access_token>`.

4. Тело запроса

   Формат: `application/json`.

   Метод поддерживает частичное обновление: в запрос можно передавать один или несколько полей профиля; не переданные поля не изменяются.

   Доступные для редактирования поля (в зависимости от модели профиля):
   - имя пользователя (string) - новое имя пользователя,
   - `email` (string, optional, формат email) — новый email пользователя.

   Правила:
   - Все переданные поля должны соответствовать требованиям к профилю (REQ_PROFILE_001 — Общие требования к профилю, REQ_PROFILE_002 — Управление email и др.).
   - Передача полей, не предусмотренных моделью профиля, должна приводить к ошибке `400 Bad Request` (если это реализовано).
   - Если тело запроса пустое или не содержит ни одного допустимого поля, метод:
     - обрабатывает запрос как отсутствие изменений (no-op),
     - возвращает `200 OK`,
     - не изменяет данные пользователя.

   Пример запроса (обновление только email):

```http
PATCH /me HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token>
Content-Type: application/json

{
  "email": "user@example.com"
}
```

5. Ответ при успехе

   - Код ответа: `200 OK`.
   - Формат: `application/json`.

   Структура ответа:

```json
{
  "id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
  "username": "string",
  "email": "string",
  "role": "admin"
}
```

   Описание полей:
   - `id` (string, uuid) — идентификатор пользователя.
   - `username` (string) — логин пользователя.
   - `email` (string) — актуальный email пользователя.
   - `role` (string) — роль пользователя (`admin`, `seller`, `user`).

   Пример успешного ответа:

```http
HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
  "username": "user",
  "email": "user@example.com",
  "role": "user"
}
```

6. Ошибки

   - `400 Bad Request` — ошибка валидации.
     - Причины:
       - `email` не соответствует формату email;
       - другие нарушения правил REQ_PROFILE_002 (если есть, например, максимальная длина).
     - Возможный формат тела ответа:

```json
{
  "code": 400,
  "message": "Validation error"
}
```

   - `401 Unauthorized` — пользователь не авторизован или `access_token` недействителен/отсутствует.

7. Особые условия

   - Метод всегда обновляет данные только текущего пользователя, определённого по `access_token`; передавать идентификатор пользователя в запросе нельзя.
   - Через этот API пользователь может самостоятельно менять email, имя, но не может менять роль пользователя.


## API_PROFILE_003 — Смена пароля текущего пользователя

1. Назначение
   - Метод предназначен для смены пароля текущего авторизованного пользователя на новый при указании корректного текущего пароля.

2. URL и HTTP‑метод
   - HTTP‑метод: POST
   - URL: /me/change-password

3. Доступ и авторизация
   - Метод доступен только авторизованным пользователям (всем ролям: admin, seller, user).
   - Запрос должен выполняться с валидным `access_token` в заголовке `Authorization: Bearer <access_token>`.

4. Тело запроса

   Формат: `application/json`  
   Схема: `ChangePasswordDto`.

   Обязательные поля:
   - `currentPassword` (string) — текущий пароль пользователя.
   - `newPassword` (string) — новый пароль пользователя (минимум 8 символов, согласно схеме).

   Правила:
   - `currentPassword` должен совпадать с текущим паролем пользователя.
   - `newPassword` должен:
     - иметь длину не менее 8 символов;
     - соответствовать требованиям REQ_AUTH_001 — Требования к регистрации.
   - Новый пароль должен отличаться от текущего:
     - если не отличается — это считается ошибкой валидации (`400 Bad Request`);

   Пример запроса:

```http
POST /me/change-password HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token>
Content-Type: application/json

{
  "currentPassword": "oldPassword123!",
  "newPassword": "newPassword456!"
}
```

5. Ответ при успехе

   - Код ответа: `200 OK`.
   - Тело ответа не описано в Swagger, считаем, что:
     - содержит сервисное сообщение об успешной смене пароля.

6. Ошибки

   - `400 Bad Request` — неверные данные или пароль.
     - Возможные причины:
       - не передано одно из обязательных полей (`currentPassword`, `newPassword`);
       - `newPassword` короче 6 символов или не соответствует требованиям безопасности;
       - `currentPassword` не совпадает с текущим паролем пользователя;
       - `newPassword` совпадает с `currentPassword`, если это запрещено.
   - `401 Unauthorized` — пользователь не авторизован или `access_token` недействителен/отсутствует.

   Возможный формат тела ошибки:

```json
{
  "code": 400,
  "message": "Invalid current password"
}
```

или

```json
{
  "code": 401,
  "message": "Unauthorized"
}
```

7. Особые условия

   - Метод изменяет пароль только для текущего пользователя, определённого по `access_token`; через этот endpoint нельзя менять пароли других пользователей.
