## API_PROFILE_001 — Получение данных текущего пользователя

1.1. Назначение: получение данных профиля текущего авторизованного пользователя по его `access_token`.

1.2. Метод и URL:
- GET /me

1.3. Доступ:
- Метод доступен только авторизованным пользователям (`admin`, `seller`, `user`).
- Требуется валидный `access_token` в заголовке `Authorization: Bearer <access_token>`.

1.4. Тело запроса:
- Отсутствует.

1.5. Успех:
- 200 OK.
- Формат ответа: application/json.
- Возвращаются данные профиля текущего пользователя.

1.5.1. Пример успешного ответа:
```json
{
  "id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
  "username": "string",
  "email": "string",
  "role": "admin"
}
```

1.5.2. Поля ответа:
- `id` — идентификатор пользователя (uuid).
- `username` — логин пользователя.
- `email` — email пользователя (string или null).
- `role` — роль пользователя (`admin`, `seller`, `user`).

1.6. Ошибки:
- 1.6.1. 401 Unauthorized — пользователь не авторизован или `access_token` отсутствует/некорректен.

1.6.2. Пример ошибки 401:
```json
{
  "code": 401,
  "message": "Missing token"
}
```

1.7. Особые условия:
- Метод всегда возвращает данные только текущего пользователя; передавать `id` пользователя в запросе нельзя.



## API_PROFILE_002 — Обновление профиля текущего пользователя

2.1. Назначение: частичное обновление данных профиля текущего авторизованного пользователя (email, имя и др. разрешённые поля).

2.2. Метод и URL:
- PATCH /me

2.3. Доступ:
- Метод доступен только авторизованным пользователям (`admin`, `seller`, `user`).
- Требуется валидный `access_token` в заголовке `Authorization: Bearer <access_token>`.

2.4. Тело запроса:
- Формат: application/json.
- Частичное обновление: передаются только изменяемые поля.

2.4.1. Доступные поля:
- `username` (string, optional) — новое имя пользователя (если предусмотрено моделью).
- `email` (string, optional) — новый email пользователя (формат email).

2.4.2. Правила:
- Все переданные поля должны соответствовать требованиям к профилю (REQ_PROFILE_001, REQ_PROFILE_002).
- Передача неизвестных полей может приводить к 400 Bad Request (если реализовано).
- Если тело запроса пустое или не содержит допустимых полей:
  - изменения не выполняются (no-op);
  - возвращается 200 OK с неизменённым профилем.

2.5. Пример запроса:
```http
PATCH /me HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token>
Content-Type: application/json

{
  "email": "user@example.com"
}
```

2.6. Успех:
- 200 OK.
- Формат ответа: application/json.
- Возвращается актуальное состояние профиля.

2.6.1. Пример успешного ответа:
```json
{
  "id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
  "username": "user",
  "email": "user@example.com",
  "role": "user"
}
```

2.6.2. Поля ответа:
- `id` — идентификатор пользователя (uuid).
- `username` — логин пользователя.
- `email` — актуальный email пользователя.
- `role` — роль пользователя (`admin`, `seller`, `user`).

2.7. Ошибки:
- 2.7.1. 400 Bad Request — ошибка валидации (некорректный email или другие нарушения REQ_PROFILE_002).
- 2.7.2. 401 Unauthorized — пользователь не авторизован или `access_token` недействителен/отсутствует.

2.7.3. Пример ошибки 400:
```json
{
  "code": 400,
  "message": "Validation error"
}
```

2.8. Особые условия:
- Метод изменяет только профиль текущего пользователя, определённого по `access_token`.
- Через этот API пользователь не может менять свою роль.



## API_PROFILE_003 — Смена пароля текущего пользователя

3.1. Назначение: смена пароля текущего авторизованного пользователя при указании корректного текущего пароля.

3.2. Метод и URL:
- POST /me/change-password

3.3. Доступ:
- Метод доступен только авторизованным пользователям (`admin`, `seller`, `user`).
- Требуется валидный `access_token` в заголовке `Authorization: Bearer <access_token>`.

3.4. Тело запроса:
- Формат: application/json.
- Схема: ChangePasswordDto.
- Обязательные поля: `currentPassword`, `newPassword`.

3.4.1. Поле currentPassword:
- Обязательно.
- Тип: string.
- Должно совпадать с текущим паролем пользователя.

3.4.2. Поле newPassword:
- Обязательно.
- Тип: string.
- Длина: минимум 8 символов (или по REQ_AUTH_001).
- Должно удовлетворять требованиям безопасности (REQ_AUTH_001).
- Должно отличаться от `currentPassword` (если так определено в правилах).

3.5. Пример запроса:
```http
POST /me/change-password HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token>
Content-Type: application/json

{
  "currentPassword": "oldPassword123!",
  "newPassword": "newPassword456!"
}
```

3.6. Успех:
- 200 OK.
- Тело ответа может содержать сервисное сообщение об успешной смене пароля (либо быть пустым — согласно Swagger/реализации).

3.7. Ошибки:
- 3.7.1. 400 Bad Request — неверные данные или нарушение правил:
  - отсутствует `currentPassword` или `newPassword`;
  - `newPassword` не соответствует требованиям безопасности;
  - `currentPassword` не совпадает с текущим паролем пользователя;
  - `newPassword` совпадает с `currentPassword` (если запрещено).
- 3.7.2. 401 Unauthorized — пользователь не авторизован или `access_token` недействителен/отсутствует.

3.7.3. Примеры ошибок:
```json
{
  "code": 400,
  "message": "Invalid current password"
}
```

```json
{
  "code": 401,
  "message": "Unauthorized"
}
```

3.8. Особые условия:
- Метод изменяет пароль только для текущего пользователя, определённого по `access_token`.
- Через этот endpoint нельзя менять пароли других пользователей.
