## API_USERS_001 — Получение списка пользователей (ADMIN, SELLER)

1.1. Назначение: получение списка всех пользователей системы (id, username, role) для администраторов и продавцов.

1.2. Метод и URL:
- GET /users

1.3. Доступ:
- Метод доступен только авторизованным пользователям с ролями `admin` и `seller`.
- Требуется валидный `access_token` в заголовке `Authorization: Bearer <access_token>`.

1.4. Параметры запроса:
- Query/path/header‑параметры не используются.

1.5. Тело запроса:
- Отсутствует.

1.6. Пример запроса:
```http
GET /users HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token>
```

1.7. Успех:
- 200 OK.
- Формат ответа: application/json.
- Тип: массив объектов пользователя.

1.7.1. Пример успешного ответа:
```json
[
  {
    "id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
    "username": "admin",
    "role": "user"
  }
]
```

1.7.2. Поля объекта пользователя:
- `id` — уникальный идентификатор пользователя (UUID, string).
- `username` — логин пользователя (string).
- `role` — роль пользователя (`user`, `admin`, `seller`).

1.8. Ошибки:
- 1.8.1. 401 Unauthorized — отсутствует или некорректный `access_token`.
- 1.8.2. 403 Forbidden — пользователь авторизован, но его роль не `admin` и не `seller`.

1.8.3. Пример ошибки 401:
```json
{
  "code": 401,
  "message": "string"
}
```

1.8.4. Пример ошибки 403:
```json
{
  "message": "string"
}
```

1.9. Особые условия:
- Метод возвращает список всех пользователей системы, без фильтрации по ролям и статусу.



## API_USERS_002 — Запрос на сброс пароля (FORGOT_PASSWORD)

2.1. Назначение: инициирование процесса сброса пароля пользователя по email (отправка письма с инструкциями).

2.2. Метод и URL:
- POST /users/forgot-password

2.3. Доступ:
- Метод доступен неавторизованным пользователям.
- `access_token` не требуется.
- Запрос выполняется без заголовка `Authorization`.

2.4. Параметры запроса:
- Query/path/header‑параметры не используются.

2.5. Тело запроса:
- Формат: application/json.
- Тип: строка, содержащая email пользователя.

2.5.1. Правила:
- Тело запроса — строка с email.
- Email должен быть в корректном формате.
- Email должен принадлежать зарегистрированному пользователю.

2.6. Пример запроса:
```http
POST /users/forgot-password HTTP/1.1
Host: api.example.com
Content-Type: application/json

"user@example.com"
```

2.7. Успех:
- 200 OK.
- Формат ответа: application/json.
- Возвращается сервисное сообщение.

2.7.1. Пример успешного ответа:
```json
{
  "message": "Письмо с инструкциями по восстановлению пароля отправлено на ваш email"
}
```

2.8. Ошибки:
- 2.8.1. 400 Bad Request — неверные данные.
- 2.8.2. 404 Not Found — пользователь с указанным email не найден.
- 2.8.3. 500 Internal Server Error — ошибка сервера при обработке запроса (например, ошибка почтового сервиса).

2.8.4. Пример ошибки 400:
```json
{
  "message": "string"
}
```

2.8.5. Пример ошибки 404:
```json
{
  "message": "string"
}
```

2.8.6. Пример ошибки 500:
```json
{
  "message": "string"
}
```

2.9. Особые условия:
- Поведение при несуществующем email (404 или всегда 200) должно быть согласовано с требованиями безопасности.
- Текст сообщений может не раскрывать, существует ли пользователь в системе (по безопасности).



## API_USERS_003 — Установка нового пароля (RESET_PASSWORD)

3.1. Назначение: установка нового пароля по токену восстановления, полученному по email после запроса `/users/forgot-password`.

3.2. Метод и URL:
- POST /users/reset-password

3.3. Доступ:
- Метод доступен неавторизованным пользователям.
- `access_token` не требуется.
- Запрос выполняется без заголовка `Authorization`.

3.4. Параметры запроса:
- Query/path/header‑параметры не используются.

3.5. Тело запроса:
- Формат: application/json.
- Тип: строка, содержащая токен восстановления пароля.

3.5.1. Правила:
- Тело запроса — строка с токеном.
- Токен должен быть корректным и действующим.
- Токен должен быть связан с существующим пользователем.

3.6. Пример запроса:
```http
POST /users/reset-password HTTP/1.1
Host: api.example.com
Content-Type: application/json

"token_string_here"
```

3.7. Успех:
- 200 OK.
- Формат ответа: application/json.
- Возвращается сервисное сообщение о успешной смене пароля.

3.7.1. Пример успешного ответа:
```json
{
  "message": "Пароль успешно изменен"
}
```

3.8. Ошибки:
- 3.8.1. 400 Bad Request — неверный, устаревший или уже использованный токен.
- 3.8.2. 404 Not Found — пользователь, связанный с токеном, не найден.
- 3.8.3. 500 Internal Server Error — внутренняя ошибка сервера при обработке запроса.

3.8.4. Пример ошибки 400:
```json
{
  "message": "string"
}
```

3.8.5. Пример ошибки 404:
```json
{
  "message": "string"
}
```

3.8.6. Пример ошибки 500:
```json
{
  "message": "string"
}
```

3.9. Особые условия:
- Токен должен быть одноразовым и иметь ограниченный срок действия.
- Конкретные правила генерации, хранения и проверки токена описываются в функциональных требованиях к восстановлению пароля.
