## API_WALLET_001 — Получение текущего баланса кошелька

1.1. Назначение: получение текущего баланса кошелька аутентифицированного пользователя (при отсутствии кошелька он создаётся с балансом 0).

1.2. Метод и URL:
- GET /wallet

1.3. Доступ:
- Метод доступен только авторизованным пользователям (`user`, `seller`, `admin`).
- Требуется валидный `access_token` в заголовке `Authorization: Bearer <access_token>`.

1.4. Параметры запроса:
- Query/path/header‑параметры не используются.

1.5. Тело запроса:
- Отсутствует.

1.6. Пример запроса:
```http
GET /wallet HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token>
```

1.7. Успех:
- 200 OK.
- Формат ответа: application/json.
- Возвращается текущий кошелёк пользователя.

1.7.1. Пример успешного ответа:
```json
{
  "id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
  "balance": 1234.5
}
```

1.7.2. Поля ответа:
- `id` — идентификатор кошелька (UUID, string).
- `balance` — текущий баланс кошелька (number, основная валюта системы).

1.8. Ошибки:
- 1.8.1. 401 Unauthorized — отсутствует или некорректный `access_token`.
- 1.8.2. 403 Forbidden — пользователь авторизован, но его роль не `user`, не `seller` и не `admin`.

1.8.3. Пример ошибки 401:
```json
{
  "code": 401,
  "message": "string"
}
```

1.8.4. Пример ошибки 403:
```json
{
  "code": 403,
  "message": "string"
}
```

1.9. Особые условия:
- При первом запросе, если у пользователя ещё нет кошелька, он создаётся автоматически с `balance = 0`.
- Баланс кошелька всегда должен быть неотрицательным (`balance >= 0`).



## API_WALLET_002 — Изменение баланса кошелька (DEPOSIT/WITHDRAW)

2.1. Назначение: изменение баланса кошелька текущего пользователя — зачисление (`DEPOSIT`) или списание (`WITHDRAW`) средств.

2.2. Метод и URL:
- PATCH /wallet

2.3. Доступ:
- Метод доступен только авторизованным пользователям (`user`, `seller`, `admin`).
- Требуется валидный `access_token` в заголовке `Authorization: Bearer <access_token>`.

2.4. Тело запроса:
- Формат: application/json.
- Обязательные поля: `operation`, `amount`.
- Необязательное поле: `description`.

2.4.1. Поле operation:
- Обязательно.
- Тип: string.
- Допустимые значения: `DEPOSIT`, `WITHDRAW`.

2.4.2. Поле amount:
- Обязательно.
- Тип: number.
- Значение: `amount > 0`.
- Максимум: 1 000 000 (максимальная сумма одной операции зачисления).

2.4.3. Поле description:
- Опциональное.
- Тип: string.
- Комментарий к операции.

2.5. Пример запроса:
```json
{
  "operation": "DEPOSIT",
  "amount": 500,
  "description": "Пополнение счета - З/П"
}
```

2.6. Успех:
- 200 OK.
- Формат ответа: application/json.

2.6.1. Пример успешного ответа:
```json
{
  "success": true,
  "balance": 734.5
}
```

2.6.2. Поля ответа:
- `success` — признак успешного выполнения операции.
- `balance` — новый баланс кошелька после операции.

2.7. Ошибки:
- 2.7.1. 400 Bad Request — неверные данные (`operation` не поддерживается, `amount <= 0`, превышен максимальный лимит и т.п.).
- 2.7.2. 401 Unauthorized — отсутствует или некорректный `access_token`.
- 2.7.3. 403 Forbidden — роль не `user`, `seller`, `admin`.
- 2.7.4. 409 Conflict — недостаточно средств для операции `WITHDRAW`.

2.7.5. Пример ошибки 409:
```json
{
  "code": 409,
  "error": "INSUFFICIENT_FUNDS",
  "message": "Недостаточно средств",
  "balance": 100
}
```

2.8. Особые условия:
- `DEPOSIT`: баланс увеличивается на `amount`.
- `WITHDRAW`: баланс уменьшается на `amount`; при `balance < amount` операция отклоняется с 409.
- Баланс кошелька всегда остаётся `>= 0`.
- Каждая операция логируется (user_id, operation, amount, старый/новый баланс, результат).



## API_WALLET_003 — История операций по кошельку

3.1. Назначение: получение списка операций по кошельку текущего пользователя с пагинацией.

3.2. Метод и URL:
- GET /wallet/transactions

3.3. Доступ:
- Метод доступен только авторизованным пользователям (`user`, `seller`, `admin`).
- Требуется валидный `access_token` в заголовке `Authorization: Bearer <access_token>`.

3.4. Query‑параметры:
- `limit` (integer, optional) — количество записей на странице, по умолчанию 10.
- `offset` (integer, optional) — смещение от начала списка, по умолчанию 0.

3.4.1. Параметр limit:
- Опциональный.
- Тип: integer.
- Минимум: 1.
- Значение по умолчанию: 10.

3.4.2. Параметр offset:
- Опциональный.
- Тип: integer.
- Минимум: 0.
- Значение по умолчанию: 0.

3.5. Пример запроса:
```http
GET /wallet/transactions?limit=10&offset=0 HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token>
```

3.6. Успех:
- 200 OK.
- Формат ответа: application/json.
- Возвращается страница операций и параметры пагинации.

3.6.1. Пример успешного ответа:
```json
{
  "items": [
    {
      "id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
      "type": "DEPOSIT",
      "amount": 500,
      "createdAt": "2026-06-17T10:54:42.325Z",
      "description": "Оплата заказа #123"
    }
  ],
  "total": 2,
  "limit": 10,
  "offset": 0,
  "hasMore": false
}
```

3.6.2. Поля ответа:
- `items` — массив операций на текущей странице.
- `total` — общее количество операций по кошельку.
- `limit` — применённый лимит.
- `offset` — применённое смещение.
- `hasMore` — признак наличия дальнейших страниц (`true`/`false`).

3.6.3. Поля операции:
- `id` — идентификатор операции (UUID, string).
- `type` — тип операции (`DEPOSIT` или `WITHDRAW`).
- `amount` — сумма операции.
- `createdAt` — дата и время операции в формате ISO 8601.
- `description` — текстовое описание (может отсутствовать).

3.7. Ошибки:
- 3.7.1. 401 Unauthorized — отсутствует или некорректный `access_token`.
- 3.7.2. 403 Forbidden — роль не `user`, `seller`, `admin`.

3.7.3. Пример ошибки 401:
```json
{
  "code": 401,
  "message": "string"
}
```

3.8. Особые условия:
- Пагинация реализуется через `limit` и `offset`, согласована с другими методами (например, `GET /products`, `GET /users`).
- Значение `hasMore = true` означает, что есть операции за пределами текущей страницы (`offset + limit < total`).
