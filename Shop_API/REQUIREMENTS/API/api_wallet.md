## API_WALLET_001 — Получение текущего баланса кошелька (USER, SELLER, ADMIN)


1. Назначение
   - Метод предназначен для получения текущего баланса кошелька аутентифицированного пользователя.
   - Если кошелька ещё нет у пользователя, он автоматически создаётся с нулевым балансом.
   - Доступно для пользователей с ролями `user`, `seller` и `admin`.


2. URL и HTTP‑метод
   - HTTP‑метод: GET
   - URL: /wallet


3. Доступ и авторизация
   - Метод доступен только авторизованным пользователям с ролями `user`, `seller` и `admin`.
   - Запрос должен выполняться с валидным `access_token` в заголовке `Authorization: Bearer <access_token>`.


4. Параметры запроса
   - Параметры запроса (query/path/header) не требуются.


   Пример запроса:


```http
GET /wallet HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token>
```


5. Тело запроса
   - Тело запроса не требуется (метод `GET` не передаёт данные).


6. Ответ при успехе


   - Код ответа: `200 OK`.
   - Формат: `application/json`.


```json
{
  "id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
  "balance": 1234.5
}
```

   Описание полей:
   - `id` (string) — уникальный идентификатор кошелька (UUID).
   - `balance` (number) — текущий баланс кошелька (в основной валюте, например, рублях).


7. Ошибки


   - `401 Unauthorized` — отсутствует или некорректный `access_token`.


```json
{
  "code": 401,
  "message": "string"
}
```

   Возможные причины:
   - отсутствует заголовок `Authorization`;
   - `access_token` недействителен, истёк или некорректно сформирован.


   - `403 Forbidden` — пользователь авторизован, но его роль не `user`, не `seller` и не `admin` (недостаточно прав).


```json
{
  "code": 403,
  "message": "string"
}
```

   Возможные причины:
   - роль пользователя не разрешена для доступа к кошельку.


8. Особые условия
   - При первом запросе `/wallet` для пользователя, у которого ещё нет кошелька, система автоматически создаёт кошелек с `balance = 0`.
   - Баланс кошелька должен быть неотрицательным числом (`balance >= 0`).
   - Правила валидации и поведения при ошибках должны быть согласованы с функциональными требованиями к модулю кошелька (REQ_WALLET_001).


## API_WALLET_002 — Изменение баланса кошелька (USER, SELLER, ADMIN)

### 1. Назначение
- Изменяет баланс кошелька: зачисляет (`DEPOSIT`) или списывает (`WITHDRAW`) сумму.
- При недостаточном балансе для `WITHDRAW` возвращается `409 Conflict`.
- Доступно для ролей: `user`, `seller`, `admin`.
- Максимальная сумма зачисления - 1000000 (миллион рублей)

### 2. URL и метод
- `PATCH /wallet`

### 3. Авторизация
- Требуется: `Authorization: Bearer <access_token>`
- Роли: `user`, `seller`, `admin`

### 4. Тело запроса (application/json)

| Field      | Type     | Required | Описание                                      |
|------------|----------|----------|-----------------------------------------------|
| `operation`| `string` | Yes      | `DEPOSIT` (зачисление) или `WITHDRAW` (списание) |
| `amount`   | `number` | Yes      | Сумма > 0                                     |
| `description`| `string`| No       | Комментарий к пополнению (опционально)                |

Пример:
```json
{
  "operation": "DEPOSIT",
  "amount": 500,
  "description": "Пополнение счета - З/П"
}
```

### 5. Ответ при успехе (200 OK)
```json
{
  "success": true,
  "balance": 734.5
}
```
- `balance` — новый баланс после операции.

### 6. Ошибки

| Код  | Описание                                              |
|------|-------------------------------------------------------|
| 400  | Неверный `operation` или `amount` <= 0               |
| 401  | Отсутствует/некорректный `access_token`              |
| 403  | Роль не `user`, `seller`, `admin`                    |
| 409  | Недостаточно средств для `WITHDRAW`                  |

Пример 409:
```json
{
  "code": 409,
  "error": "INSUFFICIENT_FUNDS",
  "message": "Недостаточно средств",
  "balance": 100
}
```

### 7. Особые условия
- `DEPOSIT`: баланс увеличивается на `amount`.
- `WITHDRAW`: баланс уменьшается на `amount`; при `balance < amount` → `409`.
- Баланс всегда `>= 0`.
- Операция логируется: `user_id`, `operation`, `amount`, `старый/новый баланс`, результат.


## API_WALLET_003 — История операций по кошельку (USER, SELLER, ADMIN)

### 1. Назначение
- Возвращает список операций (пополнения и списания) по кошельку пользователя с пагинацией.
- Доступно для ролей: `user`, `seller`, `admin`.

### 2. URL и метод
- `GET /wallet/transactions`

### 3. Авторизация
- Требуется: `Authorization: Bearer <access_token>`
- Роли: `user`, `seller`, `admin`

### 4. Query‑параметры

| Параметр  | Тип      | Default | Описание                    |
|-----------|----------|---------|-----------------------------|
| `limit`   | `integer`| 10      | Количество записей на странице |
| `offset`  | `integer`| 0       | Смещение от начала списка   |

Пример:
```http
GET /wallet/transactions?limit=10&offset=0
Authorization: Bearer <access_token>
```

### 5. Ответ при успехе (200 OK)
```json
{
  "items": [
    {
      "id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
      "type": "DEPOSIT",
      "amount": 500,
      "createdAt": "2026-06-17T10:54:42.325Z",
      "description": "Оплата заказа #123"
    }
  ],
  "total": 2,
  "limit": 10,
  "offset": 0,
  "hasMore": false
}
```

**Поля ответа:**
- `items` — массив операций на текущей странице.
- `total` — общее количество операций.
- `limit` — количество записей на странице (из запроса).
- `offset` — смещение (из запроса).
- `hasMore` — `true`, если есть следующие страницы.

**Поля операции:**
- `id` — UUID операции.
- `type` — `DEPOSIT` или `WITHDRAW`.
- `amount` — сумма операции.
- `createdAt` — время создания (ISO 8601).
- `description` — описание (опционально).

### 6. Ошибки

| Код  | Описание                                              |
|------|-------------------------------------------------------|
| 401  | Отсутствует/некорректный `access_token`              |
| 403  | Роль не `user`, `seller`, `admin`                    |

Пример 401:
```json
{
  "code": 401,
  "message": "string"
}
```

### 7. Особые условия
- Пагинация: `offset` + `limit` для навигации по записям.
- `hasMore = true` → есть следующие страницы (`offset += limit`).
- Операция логируется: `user_id`, `limit`, `offset`, `timestamp`, результат.
- Пагинация согласована с другими методами (например, `GET /users`, `GET /products`).
