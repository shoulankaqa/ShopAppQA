## API_ADMIN_001 — Создание пользователя (ADMIN)

1.1. Назначение: создание нового пользователя с ролью admin, seller или user.  

1.2. Метод и URL:
- POST /admin/users

1.3. Доступ:
- Только для пользователей с ролью admin.
- Требуется валидный токен авторизации.

1.4. Запрос:
- Content-Type: application/json.
- Обязательные поля: username, password, role.
- username должен быть уникальным.
- По умолчанию password должен соответствовать REQ_AUTH_001.
- Администратор может задать упрощённый пароль в админ-интерфейсе, если это допускается REQ_USER_002.

1.5. Пример запроса:
```http
POST /admin/users HTTP/1.1
Host: api.example.com
Authorization: Bearer <token>
Content-Type: application/json

{
  "username": "seller_02",
  "password": "Seller123!",
  "role": "seller"
}
```

1.6. Успех:
- 201 Created.
- Ответ соответствует Swagger-схеме успешного ответа пользователя.
- После успешного создания пользователь должен иметь возможность авторизоваться с указанными username и password.

1.7. Пример ответа:
```http
HTTP/1.1 201 Created
Content-Type: application/json

{
  "id": "1",
  "username": "seller_02",
  "email": null,
  "role": "seller"
}
```

1.8. Ошибки:
- 400 Bad Request — некорректные данные, отсутствуют обязательные поля, неуникальный username.
- 401 Unauthorized — отсутствует или невалиден токен.
- 403 Forbidden — у пользователя нет роли admin.
- 500 Internal Server Error — внутренняя ошибка сервера.


## API_ADMIN_002 — Удаление пользователя (ADMIN)

2.1. Назначение: удаление существующего пользователя по идентификатору.  

2.2. Метод и URL:
- DELETE /admin/users/{id}

2.3. Доступ:
- Только для пользователей с ролью admin.
- Требуется валидный токен авторизации.

2.4. Параметры запроса:
- Path-параметр id — идентификатор пользователя.

2.5. Пример запроса:
```http
DELETE /admin/users/1 HTTP/1.1
Host: api.example.com
Authorization: Bearer <token>
```

2.6. Успех:
- 204 No Content.
- Тело ответа отсутствует.
- После удаления пользователь с указанным id не должен иметь возможность авторизоваться.
- Пользователь не должен отображаться в списке пользователей.

2.7. Пример ответа:
```http
HTTP/1.1 204 No Content
```

2.8. Ошибки:
- 401 Unauthorized — отсутствует или невалиден токен.
- 403 Forbidden — у пользователя нет роли admin.
- 404 Not Found — пользователь с указанным id не найден.
- 500 Internal Server Error — внутренняя ошибка сервера.

2.9. Особое условие:
- Повторная попытка удалить уже удалённого пользователя должна возвращать 404 Not Found.


## API_ADMIN_003 — Обновление пользователя (ADMIN)

3.1. Назначение: обновление данных пользователя по идентификатору.  

3.2. Метод и URL:
- PATCH /admin/users/{id}

3.3. Доступ:
- Только для пользователей с ролью admin.
- Требуется валидный токен авторизации.

3.4. Параметры запроса:
- Path-параметр id — идентификатор пользователя.

3.5. Запрос:
- Content-Type: application/json.
- Обновляются только переданные поля.
- Допустимые поля: username, password, role.
- username должен быть уникальным.
- password по умолчанию должен соответствовать REQ_AUTH_001.
- Администратор может задать упрощённый пароль в соответствии с REQ_USER_002.
- role может принимать только значения admin, seller, user.
- Лишние поля запрещены.

3.6. Пример запроса:
```http
PATCH /admin/users/1 HTTP/1.1
Host: api.example.com
Authorization: Bearer <token>
Content-Type: application/json

{
  "role": "user"
}
```

3.7. Успех:
- 200 OK.
- Ответ содержит обновлённого пользователя.
- Если в запросе не передано ни одного поля, запрос должен обработаться успешно и вернуть 200 OK без изменения данных.
- После изменения username и/или password пользователь должен иметь возможность авторизоваться с новыми значениями.
- Изменение role должно вступать в силу сразу после сохранения.

3.8. Пример ответа:
```http
HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 8,
  "username": "seller_02",
  "role": "user"
}
```

3.9. Ошибки:
- 400 Bad Request — некорректные данные, лишние поля, неуникальный username, недопустимое значение role.
- 401 Unauthorized — отсутствует или невалиден токен.
- 403 Forbidden — у пользователя нет роли admin.
- 404 Not Found — пользователь не найден.
- 500 Internal Server Error — внутренняя ошибка сервера.
