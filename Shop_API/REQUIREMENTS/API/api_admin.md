## API_ADMIN_001 — Создание пользователя (ADMIN)

1. Назначение
   - Метод предназначен для создания нового пользователя с указанной ролью (`admin`, `seller`, `user`).

2. URL и HTTP‑метод
   - HTTP‑метод: POST
   - URL: /admin/users

3. Доступ и авторизация
   - Метод доступен только пользователям с ролью `admin`.
   - Запрос должен выполняться с валидным токеном авторизации (см. REQ_AUTH_xxx).

4. Тело запроса

   Формат: `application/json`  
   Состав и типы полей соответствуют Swagger‑схеме `UserCreate` (username, password, role).

   Обязательные поля:
   - `username` (string) — логин пользователя, должен быть уникален в системе.
   - `password` (string) — пароль пользователя.
   - `role` (string) — роль пользователя, одно из значений: `admin`, `seller`, `user`.

   Правила:
   - По умолчанию пароль должен соответствовать требованиям REQ_AUTH_002 — Требования к паролю.
   - Администратор может **осознанно** создать пользователя с паролем, не соответствующим REQ_AUTH_002 (см. REQ_USER_006 — Особые права администратора).
   - Уникальность `username` — обязательна и не может быть нарушена.

5. Ответ при успехе

   - Код ответа: `201 Created`
   - Формат: `application/json`
   - Структура ответа соответствует Swagger‑схеме успешного ответа:
     - `id` (string) — уникальный идентификатор пользователя (UUID).
     - `username` (string) — логин пользователя.
     - `email` (string) — email пользователя (может быть пустым/null, если не задаётся при создании).
     - `role` (string) — роль пользователя (`admin`, `seller`, `user`).

6. Ошибки

   - `400 Bad Request` — неверные данные запроса (например, отсутствуют обязательные поля, некорректные форматы, неуникальный `username`).
     - Тело ответа (application/json):
       - `code` (number) — код ошибки (400).
       - `message` (string) — описание ошибки.
   - `401 Unauthorized` — при отсутствии или недействительном токене авторизации.
   - `403 Forbidden` — если запрос выполняет пользователь без роли `admin`.
   - `500 Internal Server Error` — при внутренней ошибке сервера.

7. Особые условия

   - При успешном создании пользователь должен иметь возможность авторизоваться с указанными `username` и `password`.
   - При попытке создать пользователя с уже существующим `username` система должна вернуть `400 Bad Request` с сообщением о неуникальном логине.
   - Использование администратором пароля, не соответствующего REQ_AUTH_002, не блокирует создание пользователя, но считается исключением по REQ_USER_006.

8. Пример запроса

```http
POST /admin/users HTTP/1.1
Host: api.example.com
Authorization: Bearer <token>
Content-Type: application/json

{
  "username": "new_user_1",
  "password": "Qwerty123!",
  "role": "seller"
}
```

9. Пример успешного ответа

```http
HTTP/1.1 201 Created
Content-Type: application/json

{
  "id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
  "username": "new_user_1",
  "email": "",
  "role": "seller"
}
```

## API_ADMIN_002 — Удаление пользователя (ADMIN)

1. Назначение
   - Метод предназначен для удаления существующего пользователя по идентификатору.

2. URL и HTTP‑метод
   - HTTP‑метод: DELETE
   - URL: /admin/users/{id}

3. Доступ и авторизация
   - Метод доступен только пользователям с ролью `admin`.
   - Запрос должен выполняться с валидным токеном авторизации (см. REQ_AUTH_xxx).

4. Параметры запроса

   Path‑параметры:
   - `id` (integer, required) — идентификатор пользователя, который должен быть удалён.

5. Ответ при успехе

   - Код ответа: `204 No Content`.
   - Тело ответа отсутствует.
   - После успешного удаления:
     - пользователь с указанным `id` не должен иметь возможность авторизоваться в системе;
     - пользователь не должен отображаться в списке пользователей (см. REQ_USER_004).

6. Ошибки

   - `403 Forbidden` — при отсутствии прав доступа (запрос выполняет пользователь без роли `admin`).
     - Тело ответа (application/json):
       - `message` (string) — краткое описание причины отказа.
   - `404 Not Found` — если пользователь с указанным `id` не найден.
     - Тело ответа (application/json):
       - `code` (number) — код ошибки (404).
       - `message` (string) — описание ошибки (например, "User not found").
   - `401 Unauthorized` — при отсутствии или недействительном токене авторизации (общее правило для защищённых методов).
   - `500 Internal Server Error` — при внутренней ошибке сервера.

7. Особые условия

   - При повторной попытке удалить уже удалённого пользователя метод должен возвращать `404 Not Found`.
   - Удаление пользователя должно быть необратимым в рамках текущей реализации (если используется физическое удаление); при логическом удалении поведение и статус описываются отдельно в REQ_USER_004.

   
   
## API_ADMIN_003 — Обновление пользователя (ADMIN)

1. Назначение
   - Метод предназначен для обновления данных пользователя (логин, роль и при необходимости пароль) по идентификатору.

2. URL и HTTP‑метод
   - HTTP‑метод: PATCH
   - URL: /admin/users/{id}

3. Доступ и авторизация
   - Метод доступен только пользователям с ролью `admin`.
   - Запрос должен выполняться с валидным токеном авторизации (см. REQ_AUTH_xxx).

4. Параметры запроса

   Path‑параметры:
   - `id` (integer, required) — идентификатор пользователя, данные которого обновляются.

5. Тело запроса

   Формат: `application/json`  
   Обновляются только указанные поля; если поле не передано, оно остаётся без изменений.

   Поля:
   - `username` (string, optional) — новый логин пользователя.
     - Должен быть уникален в системе.
   - `password` (string, optional) — новый пароль пользователя.
     - При указании нового пароля по умолчанию он должен соответствовать требованиям REQ_AUTH_002.
     - Администратор может задать пароль, не соответствующий REQ_AUTH_002, в соответствии с REQ_USER_006 — Особые права администратора.
   - `role` (string, optional) — новая роль пользователя, одно из значений: `admin`, `seller`, `user`.

   Правила:
   - Передача полей, не описанных в схеме (лишние поля), должна приводить к ошибке `400 Bad Request`.
   - Значение `role` вне перечисленных значений должно приводить к ошибке `400 Bad Request`.

6. Ответ при успехе

   - Код ответа: `200 OK`.
   - Формат: `application/json`.
   - Тело ответа содержит обновлённого пользователя:
     - `id` (integer) — идентификатор пользователя.
     - `username` (string) — актуальный логин пользователя.
     - `role` (string) — актуальная роль пользователя (`admin`, `seller`, `user`).

7. Ошибки

   - `400 Bad Request` — некорректные данные запроса:
     - лишние поля;
     - неверное значение `role`;
     - неуникальный `username`;
     - другие ошибки формата данных.
     - Тело ответа (application/json):
       - `code` (number) — код ошибки (400);
       - `message` (string) — описание ошибки.
   - `401 Unauthorized` — отсутствует или невалидный access‑токен.
     - Тело ответа:
       - `code` (401),
       - `message` (string).
   - `403 Forbidden` — доступ запрещён (роль не `admin`).
     - Тело ответа:
       - `code` (403),
       - `message` (string).
   - `404 Not Found` — пользователь не найден или указан некорректный `id`.
     - Тело ответа:
       - `code` (404),
       - `message` (string).
   - `500 Internal Server Error` — внутренняя ошибка сервера.

8. Особые условия

   - Если в запросе не передано ни одного поля (`username`, `password`, `role`), система всё равно должна:
     - корректно обработать запрос,
     - вернуть `200 OK`,
     - не изменять данные пользователя.
   - При успешном изменении `username` и/или `password` пользователь должен иметь возможность авторизоваться с новыми значениями.
   - Изменение `role` вступает в силу с момента сохранения и влияет на права пользователя при следующих запросах.

9. Пример запроса

```http
PATCH /admin/users/3 HTTP/1.1
Host: api.example.com
Authorization: Bearer <token>
Content-Type: application/json

{
  "username": "admin2",
  "password": "NewP@ssw0rd",
  "role": "seller"
}
```

10. Пример успешного ответа

```http
HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 3,
  "username": "admin2",
  "role": "seller"
}
```

11. Пример ответа при ошибке (400 Bad Request, занятый username)

```http
HTTP/1.1 400 Bad Request
Content-Type: application/json

{
  "code": 400,
  "message": "Username already exists"
}
```
