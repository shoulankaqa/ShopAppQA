# REQ_AUTH_001 — Требования к регистрации

Это требование описывает правила валидации полей и процесс создания новой учётной записи пользователя через форму регистрации.

## 1. Валидация логина

1.1. Логин должен содержать минимум 6 символов и максимум 32 символа.  
1.2. Логин может состоять из:
- латинских букв нижнего регистра `[a-z]`;
- цифр `[0-9]`;
- нижнего подчёркивания `_`.

1.3. Логин должен начинаться с буквы.  
1.4. Поле логина является обязательным для заполнения.  
1.5. Логин должен быть уникальным в системе.  
1.6. Запрещена регистрация с логинами, начинающимися на `admin`, `user` или `seller`.  
1.7. При вводе более 32 символов в поле логина дополнительные символы не должны вводиться: поле ограничивает ввод и не принимает символы сверх максимальной длины (например, при попытке ввода 33-го символа он не отображается в поле).

## 2. Валидация пароля

2.1. Пароль должен содержать минимум 8 символов и максимум 64 символа.  
2.2. Пароль должен включать в себя:
- только латинские буквы (A–Z, a–z), цифры (0–9) и символы `($!^@?)`;
- хотя бы одну заглавную букву (A–Z);
- хотя бы одну цифру (0–9);
- хотя бы один спецсимвол `($!^@?)`;
- минимум 4 буквы всего.

2.3. Поле пароля является обязательным для заполнения.  
2.4. Поле ввода пароля должно иметь иконку для переключения видимости ("показать/скрыть"), которая:
- доступна с клавиатуры;
- имеет видимый фокус.

2.5. При вводе более 64 символов в поле пароля дополнительные символы не должны вводиться: поле ограничивает ввод и не принимает символы сверх максимальной длины (например, при попытке ввода 65-го символа он не отображается в поле).

## 3. Процесс регистрации

3.1. Пользователь должен иметь возможность зарегистрироваться, заполнив поля логина и пароля.  
3.2. После успешной регистрации пользователь должен быть автоматически авторизован и перенаправлен в раздел "Товары".  
3.3. При успешной регистрации система должна отображать сообщение:  
`"Успех. Зарегистрировались и авторизовались под: <имя пользователя>"`.

3.4. При попытке зарегистрировать существующего пользователя система должна отобразить сообщение:  
`"Пользователь <имя пользователя> уже существует"`.

3.5. При наличии ошибок валидации система должна:
- не создавать пользователя;
- не выполнять авторизацию;
- отображать сообщения об ошибках рядом с соответствующими полями и/или общие сообщения в зависимости от состояния полей.

Логика отображения сообщений:

- Если **логин пустой** и **пароль пустой**:  
  - система отображает общее сообщение:  
    `"Требуется имя пользователя и пароль"`.

- Если **логин невалидный по формату/длине** и **поле пароля пустое**:  
  - система также отображает общее сообщение:  
    `"Требуется имя пользователя и пароль"`.

- Если **логин невалидный** (нарушены правила длины/формата), а **пароль заполнен** (неважно, валиден он или нет):  
  - система должна в первую очередь отобразить ошибку логина:  
    `"Логин должен содержать от 6 до 32 символов (a–z, 0–9, _), начинаться с буквы"`;

- Если логин начинается с зарезервированного префикса `admin`, `user` или `seller`:  
  - система отображает сообщение:  
    `"Логин не должен начинаться с admin, user или seller"`.

- Для поля пароль при нарушении требований (длина, состав, наличие заглавной буквы, цифры и спецсимвола):  
  - система отображает сообщение:  
    `"Пароль должен содержать минимум 8 символов, включая хотя бы 1 заглавную букву, 1 цифру, 1 спецсимвол ($!^@?) и минимум 4 буквы"`.

3.6. Форма регистрации должна быть полностью доступна для навигации с клавиатуры, а все активные элементы должны иметь видимый индикатор фокуса.  
3.7. Новая учётная запись по умолчанию создаётся с ролью обычного пользователя.  
3.8. Роли `"admin"` и `"seller"` не могут быть назначены через форму саморегистрации.
