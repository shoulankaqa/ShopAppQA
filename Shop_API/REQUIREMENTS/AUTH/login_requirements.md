
## REQ_AUTH_002 — Авторизация пользователя

### Основные функции

1. Пользователь должен иметь возможность авторизоваться с использованием валидных учетных данных (логин и пароль)
2. Форма авторизации должна содержать:
   - поле для логина,
   - поле для пароля,
   - кнопку "Войти"

### Обработка ошибок

3. При вводе неверных учетных данных система должна отображать универсальное сообщение, которое не раскрывает, что именно неверно, например: "Неверные учетные данные"
4. Система не должна показывать, существует ли пользователь в системе (защита от перебора логинов)
?5. Если пароль введен неверно несколько раз подряд, система должна:
   - показать сообщение о блокировке на время,
   - ограничить количество попыток авторизации (3–5 попыток в час для одного пользователя или IP)

### Безопасность

6. После успешной авторизации система должна:
   - создать сессию пользователя,
   - использовать защищенный канал связи (HTTPS) для передачи учетных данных
?7. Сессия пользователя должна иметь ограниченный срок жизни (2 часа без активности)
8. После истечения срока сессии пользователь должен быть деавторизован и перенаправлен на страницу авторизации при попытке доступа к защищенным разделам

### UX и доступность

9. После успешной авторизации пользователь должен быть перенаправлен в раздел "Товары"
10. После успешной авторизации система должна отображать сообщение: "Успех. Авторизовались под: <имя пользователя>"
11. Форма авторизации должна быть доступна для навигации с клавиатуры и иметь видимый фокус на активных элементах
12. Сообщения об ошибках должны быть читаемыми и доступными для пользователей с нарушениями зрения (например, с использованием ARIA-атрибутов)
?13. Поле пароля может иметь кнопку "Показать пароль" для удобства пользователя при вводе, при этом содержимое пароля не должно логироваться или кешироваться в открытом виде

### Дополнительные сценарии

14. Если пользователь превысил допустимое количество попыток авторизации, система должна показать сообщение: "Превышено количество попыток входа. Пожалуйста, попробуйте позже." и временно блокировать дальнейшие попытки

## REQ_AUTH_003 — Выход из системы

### Основные функции

1. Пользователь должен иметь возможность выйти из системы через доступный элемент интерфейса (через кнопку "Выйти")
2. Кнопка "Выйти" должна быть доступна во всех защищенных разделах системы

### Обработка сеанса

3. После выхода пользователь должен быть перенаправлен на страницу авторизации
4. После выхода пользователь не должен иметь доступ к защищенным разделам системы без повторной авторизации
5. После выхода сессия пользователя должна быть завершена, а данные сессии признаны недействительными (токен сессии должен быть деактивирован)
6. При попытке открыть защищенную страницу после выхода (например, через историю браузера) пользователь должен быть перенаправлен на страницу авторизации

### Безопасность

7. При выходе из системы должны завершаться все критичные действия, связанные с учетной записью (операции с корзиной, личными данными, кошельком)
8. Система поддерживает работу в нескольких вкладках браузера в рамках ыодной активной пользовательской сессии:
   - при авторизации под другим пользователем в этом же браузере/профиле текущая сессия заменяется новой;
   - все открытые вкладки начинают работать от имени последнего авторизованного пользователя;
   - при выходе из системы в одной вкладке все вкладки, использующие эту сессию, считаются разлогиненными.

### UX и доступность

9. Элемент "Выйти" должен быть доступен для навигации с клавиатуры и иметь видимый фокус.

### Дополнительные сценарии

10. Если пользователь закрывает браузер без явного выхода, система должна завершить сессию после истечения настроенного времени жизни сессии.
