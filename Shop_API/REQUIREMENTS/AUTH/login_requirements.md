# REQ_AUTH_002 — Авторизация и выход из системы

## 1. Общие положения

Это требование описывает правила валидации полей, процесс авторизации и выхода пользователя из системы.

## 2. Требования к авторизации

### 2.1. Основные функции

2.1.1. Пользователь должен иметь возможность авторизоваться с использованием валидных учётных данных (логин и пароль).  
2.1.2. Форма авторизации должна содержать:
- поле для логина;
- поле для пароля;
- кнопку "Войти".

### 2.2. Валидация и обработка ошибок

2.2.1. Поля логина и пароля являются обязательными для заполнения.  

2.2.2. Если **хотя бы одно из полей** (логин или пароль) пустое при попытке авторизации, система должна:
- не отправлять запрос на сервер;
- отображать сообщение:
  `"Требуется имя пользователя и пароль"`. 

2.2.3. Если оба поля заполнены, но учётные данные неверны (логин/пароль не совпадают с данными в системе), система должна:
- не раскрывать, что именно неверно (логин или пароль);
- отображать универсальное сообщение:
  `"Неверные учетные данные"`.

2.2.4. Система не должна показывать, существует ли пользователь в системе (защита от перебора логинов).

2.2.5. Если пароль введён неверно несколько раз подряд, система должна:
- показать сообщение о временной блокировке (`"Превышено количество попыток входа. Пожалуйста, попробуйте позже.
 "`);  
- ограничить количество попыток авторизации (5 попыток за 15 минут для одного пользователя);
- после истечения 15 минут с момента последней неверной попытки входа разблокировать пользователя с возможность авторизации (при вводе корректного пароля вход должен быть успешным).

### 2.3. Безопасность

2.3.1. После успешной авторизации система должна создать сессию пользователя.  
2.3.2. Система должна использовать защищённый канал связи (HTTPS) для передачи учётных данных.

### 2.4. UX и доступность

2.4.1. После успешной авторизации пользователь должен быть перенаправлен в раздел "Товары".  
2.4.2. После успешной авторизации система должна отображать сообщение:  
`"Успех. Авторизовались под: <имя пользователя>"`.  
2.4.3. Форма авторизации должна быть доступна для навигации с клавиатуры и иметь видимый фокус на активных элементах.  
2.4.4. Сообщения об ошибках должны быть читаемыми и доступными для пользователей с нарушениями зрения (например, с использованием ARIA-атрибутов).  
2.4.5. Поле пароля должно иметь кнопку "Показать пароль" для удобства пользователя при вводе.

## 3. REQ_AUTH_003 — Требования к выходу из системы

### 3.1. Основные функции

3.1.1. Пользователь должен иметь возможность выйти из системы через доступный элемент интерфейса (кнопка "Выйти").  
3.1.2. Кнопка "Выйти" должна быть доступна во всех защищённых разделах системы.

### 3.2. Обработка сеанса

3.2.1. После выхода пользователь должен быть перенаправлен на страницу авторизации.  
3.2.2. После выхода пользователь не должен иметь доступ к защищённым разделам системы без повторной авторизации.  
3.2.3. После выхода сессия пользователя должна быть завершена, а данные сессии признаны недействительными (токен сессии должен быть деактивирован).  
3.2.4. При попытке открыть защищённую страницу после выхода (например, через историю браузера) пользователь должен быть перенаправлен на страницу авторизации.

### 3.3. Безопасность

3.3.1. При выходе из системы должны завершаться все критичные действия, связанные с учётной записью (операции с корзиной, личными данными, кошельком).

### 3.4. Работа в нескольких вкладках

3.4.1. Система поддерживает работу в нескольких вкладках браузера.  
3.4.2. Каждая вкладка использует свою собственную сессию и работает от имени пользователя, авторизованного в этой вкладке.  
3.4.3. Выход из системы в одной вкладке не должен разлогинивать пользователя в других вкладках, где используются другие активные сессии.

### 3.5. UX и доступность

3.5.1. Элемент "Выйти" должен быть доступен для навигации с клавиатуры и иметь видимый фокус.

### 3.6. Дополнительные сценарии

3.6.1. Если пользователь закрывает браузер без явного выхода, система должна завершить сессию после истечения настроенного времени жизни сессии.
