# REQ_WALLET_001 — Кошелёк пользователя

## Основные функции

### 1. Отображение баланса

Система должна отображать текущий баланс кошелька пользователя в формате `999 999,99`  
(с пробелами в качестве разделителей тысяч и запятой в качестве десятичного разделителя).

### 2. Отображение количества операций

Система должна отображать общее количество совершённых операций по кошельку.  
Это число является общим для всех фильтров и всегда показывает полный объём истории.

### 3. Просмотр истории

Пользователь должен иметь возможность просматривать историю операций кошелька (пополнения и списания).

### 4. Фильтрация операций

Пользователь должен иметь возможность фильтровать операции по типу:

- "Все операции"
- "Пополнения"
- "Списания"

### 5. Настройка количества записей

Пользователь должен иметь возможность выбирать количество отображаемых записей на странице:  
`5`, `10`, `15`, `20`, `50`, `100`.  
По умолчанию устанавливается значение `10`.

### 6. Пагинация

Если количество операций превышает выбранное число на странице, система должна отображать пагинацию:

- кнопку **"След."** для перехода на следующую страницу;
- кнопку **"Пред."** для перехода на предыдущую страницу;
- информацию о текущей и общей страницах истории (например: "Страница 2 из 5").

## Форма пополнения баланса

На странице кошелька должна быть форма пополнения баланса, содержащая:

- поле **"Сумма пополнения"** (обязательное);
- поле **"Комментарий к пополнению"** (необязательное);
- кнопку **"Пополнить"**.

При успешном пополнении баланса система должна:

- увеличить текущий баланс на указанную сумму;
- добавить запись в историю операций с типом "Пополнение", суммой и комментарием;
- отобразить сообщение об успешном пополнении (тост-уведомление) без перезагрузки страницы.

## Валидация суммы пополнения

- Поле **"Сумма пополнения"** является обязательным для заполнения.
- Сумма пополнения должна быть строго больше `0`.
- Сумма пополнения не может начинаться с десятичного разделителя (например, `,50` — недопустимо).  
  Минимальный формат — одна цифра до разделителя (например: `5`, `5,0`).
- Система должна поддерживать ввод суммы с точностью до двух знаков после запятой.

При вводе значения с более чем двумя знаками после запятой (например, `123,133`) система:

- не изменяет введённое пользователем значение автоматически;
- отображает подсказку о том, что допустимы только два знака после запятой;
- предлагает пользователю несколько возможных вариантов округления  
  (например, `123,13`, `123,14`) в виде текстовой подсказки как рекомендацию.
- Пользователь самостоятельно вносит исправленное значение в поле "Сумма пополнения".  
  Система не подставляет округлённое значение автоматически и не изменяет поле без явного действия пользователя.

Система должна приводить уже введённое пользователем корректное значение к формату с двумя знаками после запятой при отображении, например:

- `100` → `100,00`
- `100,5` → `100,50`

### 16. Максимальная сумма пополнения

Максимальная сумма одного пополнения не может превышать **1 000 000,00** (один миллион) рублей.

### 17. Попытка пополнения выше лимита

При попытке пополнить кошелёк на сумму, превышающую **1 000 000,00** руб.:

- система должна блокировать операцию пополнения;
- система должна выводить пользователю сообщение:  
  **"Сумма пополнения не может превышать 1 000 000 рублей"**.

### 18. Некорректный ввод числа

При попытке ввода числа, которое по формату или длине не может быть корректно приведено к числу в диапазоне от **1,00** до **1 000 000,00**, система должна:

- не допускать ввод лишних символов и/или
- немедленно отображать сообщение об ошибке валидации.

## Валидация комментария

### 19. Необязательность поля

Поле **"Комментарий к пополнению"** является необязательным.

### 20. Тип содержимого

Поле должно поддерживать ввод произвольного текста (включая буквы, цифры, спецсимволы и разные языки).

### 21. Лимит длины

Максимальная длина комментария составляет **255 символов**.

### 22. Превышение лимита

При превышении лимита система должна:

- блокировать отправку формы;
- отображать сообщение: **"Максимальная длина комментария — 255 символов"**.

## Обработка ошибок

### Валидация на клиенте

Все ошибки валидации должны обрабатываться на клиентской части. При их возникновении система должна:

- отображать понятное сообщение об ошибке рядом с соответствующим полем;
- не выполнять операцию пополнения;
- не менять текущий баланс.

### Технические сбои

При технических сбоях (ошибки сети, недоступность сервера) система должна:

- перехватывать ошибку;
- вместо стандартной страницы 500 отображать пользователю сообщение:  
  **"Произошла ошибка при обработке запроса. Пожалуйста, попробуйте позже."**;
- не допускать падения интерфейса.

## UX и доступность (Accessibility)

- Поля **"Сумма пополнения"** и **"Комментарий к пополнению"** должны содержать плейсхолдер-подсказку, например:
  - для суммы: "Например, 500";
  - для комментария: "Например, пополнение для теста".

- При срабатывании валидации по количеству знаков после запятой система должна:
  - отображать текстовую рекомендацию с примером возможного округления  
    (например: "Введите допустимое значение. Ближайшие допустимые значения: 123,13 и 123,14");
  - предоставлять пользователю возможность самостоятельно исправить значение.

Все интерактивные элементы (кнопки, ссылки, поля ввода) должны:

- быть доступны для навигации с клавиатуры (`tabindex`);
- иметь видимый индикатор фокуса.

Сообщения об ошибках и статусы операций должны быть связаны с соответствующими полями через атрибуты `aria-describedby` для корректной работы скринридеров.

## Дополнительные сценарии

### Изменение фильтра операций

При изменении фильтра операций ("все", "пополнения", "списания"):

- список операций должен обновляться мгновенно;
- пагинация должна пересчитываться в соответствии с новым набором данных.

### Изменение количества элементов на странице

При изменении количества элементов на странице система должна возвращать пользователя на первую страницу списка операций.

### 31. Пополнение на максимальную сумму

Операция пополнения на максимально допустимую сумму (**1 000 000,00**) должна:

- выполняться успешно;
- корректно отражаться в истории операций и в текущем балансе.

### 32. Оплата корзины с кошелька

1. При оплате корзины с использованием кошелька:
   - сумма заказа списывается с текущего баланса кошелька пользователя;
   - текущий баланс уменьшается на сумму оплаченной корзины.
2. Если текущего баланса недостаточно для полной оплаты корзины:
   - операция оплаты кошельком не должна выполняться;
   - система должна отобразить сообщение (например):  
     **"Недостаточно средств на кошельке для оплаты корзины"**.
3. При успешной оплате корзины:
   - в истории операций кошелька должна создаваться запись с типом "Списание" на сумму заказа;
   - запись должна содержать дату, сумму и комменарий.






# REQ_WALLET_001 — Кошелёк пользователя

## 1. Общие положения

Это требование описывает функциональность пользовательского кошелька, включая отображение информации, пополнение баланса, просмотр истории операций и правила валидации.

## 2. Функциональные требования

### 2.1. Отображение информации

1. Система должна отображать текущий баланс кошелька в формате `999 999,99`

2. Система должна отображать общее количество совершённых операций по кошельку.

2. Пользователь должен иметь возможность фильтровать операции по типу:
   - "Все операции";
   - "Пополнения";
   - "Списания".
3. Пользователь должен иметь возможность настраивать количество записей на странице: `5`, `10`, `15`, `20`, `50`, `100`.  
   По умолчанию — `10`.
4. При превышении количества записей на странице должна отображаться пагинация с:
   - кнопкой **"След."**;
   - кнопкой **"Пред."**;
   - информацией о текущей странице (например: "Страница 2 из 5").

### 2.3. Пополнение баланса

1. На странице кошелька должна быть форма пополнения, содержащая:
   - поле **"Сумма пополнения"** (обязательное);
   - поле **"Комментарий к пополнению"** (необязательное);
   - кнопку **"Пополнить"**.
2. При успешном пополнении система должна:
   - увеличить баланс на указанную сумму;
   - добавить запись в историю с типом "Пополнение";
   - отобразить тост-уведомление об успехе без перезагрузки страницы.

## 3. Валидация формы пополнения

### 3.1. Валидация суммы

1. Поле **"Сумма пополнения"** обязательно для заполнения.
2. Сумма должна быть строго больше `0`.
3. Сумма не может начинаться с десятичного разделителя (например, `,50` — недопустимо).  
   Минимальный формат — одна цифра до разделителя (например: `5`, `5,0`).
4. Поддерживается точность до двух знаков после запятой.
5. При вводе более двух знаков после запятой (например, `123,133`) система должна:
   - не изменять введённое значение автоматически;
   - отображать подсказку о лимите знаков;
   - предлагать варианты округления (например, `123,13`, `123,14`) как рекомендацию;
   - требовать, чтобы пользователь самостоятельно исправил значение.
6. Корректно введённые значения должны приводиться к формату с двумя знаками:
   - `100` → `100,00`;
   - `100,5` → `100,50`.
7. Максимальная сумма пополнения — **1 000 000,00** руб.

### 3.2. Валидация комментария

1. Поле **"Комментарий к пополнению"** является необязательным.
2. Поддерживается ввод произвольного текста (буквы, цифры, спецсимволы).
3. Максимальная длина комментария — **255 символов**.

## 4. Обработка ошибок и доступность

### 4.1. Валидация

1. Все ошибки валидации обрабатываются на клиенте.
2. При ошибке система должна:
   - отображать понятное сообщение рядом с полем;
   - не выполнять операцию пополнения;
   - не менять текущий баланс.

     **"Произошла ошибка при обработке запроса. Пожалуйста, попробуйте позже."**.

1. Поля ввода должны иметь плейсхолдеры:

   - для комментария: "Например, пополнение для теста".

## 5. Дополнительные сценарии

### 5.1. Изменение параметров просмотра

1. При изменении фильтра или количества записей на странице система должна возвращать пользователя на первую страницу истории.
