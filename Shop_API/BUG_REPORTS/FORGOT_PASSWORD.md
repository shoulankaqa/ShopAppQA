# BUG_FORGOTPWD_001 (Исправлен)

## Заголовок
Email отображается в сообщении подтверждения восстановления пароля 

## Описание
На экране после заполнения формы «Забыли пароль?» отображается сообщение с конкретным email:
 - Поп-ап: "Успех! Инструкции по восстановлению пароля отправлены на вашу почту"
 - Основное сообщение: "Проверьте почту. Мы отправили инструкции по восстановлению пароля на адрес <указанная почта>"
Если email не привязан к указанному логину, письмо не придет, но пользователь видит "успех" и конкретный email, что:
 - создает ложное ожидаение письма
 - раскрывает введенный email в тексте сообщения (небезопасно для email enumeration / disclosure)
 - Нарушает REQ_AUTH_004 пункт 7 и 12           

 ## Тестовые данные
1. Логин: nonexistuser - несуществующий пользователь
2. Email: nonexistuseremail@test.com - несуществующая почта 

## Шаги воспроизведения
1. Открыть страницу http://localhost:3001/login
2. Нажать "Забыли пароль?"
3. Ввести логин в поле "Имя пользователя" из тестовых данных
4. Ввести email в поле "Email" из тестовых данных
5. Нажать кнопку "Отправить инструкции"
6. Проверить отображаемые сообщения (поп-ап + основное)

## Ожидаемый результат
Сообщение должно быть универсальным, без показа конкретного email:
 - "Если email привязан к логину, письмо с инструкциями по восстановлению пароля будет отправлено. Проверьте почтовый ящик"

## Фактический результат
1. Пользователь видит "Успех!" и конкретный email в сообщении
2. Письмо не приходит, но пользователь не знает об этом

## Серьёзность
High

## Приоритет
High

## Окружение
- Shop App: 1.0.0
- ОС: macOS 26.2
- Браузер: Comet
- URL: http://localhost:3001

# BUG_FORGOTPWD_002 (Исправлен)

## Заголовок
Сообщения в DevTools и на экране не совпадают при восстановлении пароля

## Описание
Сообщение в DevTools (network) и сообщение на экране различаются:
- DevTools:
"Если пользователь с такими данными существует, на указанный email будет отправлено письмо с инструкциями"

 - Экран (поп-ап):
"Успех! Инструкции по восстановлению пароля отправлены на вашу почту"

 - Экран (основное сообщение):
"Проверьте почту. Мы отправили инструкции по восстановлению пароля на адрес <указанный email>"

## Тестовые данные
1. Логин: nonexistuser - несуществующий пользователь
2. Email: nonexistuseremail@test.com - несуществующая почта 

## Шаги воспроизведения
1. Открыть страницу http://localhost:3001/login
2. Нажать "Забыли пароль?"
3. Ввести логин в поле "Имя пользователя" из тестовых данных
4. Ввести email в поле "Email" из тестовых данных
5. Нажать кнопку "Отправить инструкции"
6. Открыть DevTools (Network) и посмотреть ответ сервера

# BUG_FORGOTPWD_003 (Исправлен)

## Заголовок
Отсутствие ограничения частоты запросов восстановления пароля

## Описание
Система не ограничивает количество запросов восстановления пароля на один email/логин:
 - Пользователь может отправить форму восстановления пароля множество раз подряд
В результате:
 - на email приходят множество писем с токенами восстановления
 - это создает нагрузку на почтовый сервер
 - повышает риск спам-атак и сбора базы email-адресов
 - нарушает требование безопасности REQ_AUTH_004, пункт 13 (ограничение частоты запросов / rate limiting)

## Предусловия
1. Существует пользователь tuser с подтвержденным существующим email tuser@test.com

## Тестовые данные
1. Логин: tuser 
2. Email: tuser@test.com

## Шаги воспроизведения
1. Открыть страницу http://localhost:3001/login
2. Нажать "Забыли пароль?"
3. Ввести логин в поле "Имя пользователя" из тестовых данных
4. Ввести email в поле "Email" из тестовых данных
5. Нажать кнопку "Отправить инструкции"
6. Обновить страницу
7. Повторить шаги 3–6 10 раз подряд в течение нескольких минут
8. Проверить, приходят ли письма на каждый запрос и нет ли ограничения частоты запросов

## Ожидаемый результат
1. Система ограничивает частоту запросов восстановления пароля:
 - максимум 3 запроса в час на один email 
2. После превышения лимита система отображет сообщение:
"Вы слишком часто запрашиваете восстановление пароля. Пожалуйста, подождите до следующего часа."
3. Новые запросы блокируются до истечения времени

## Фактический результат
1. На каждый запрос приходит новое письмо с токеном восстановления
2. Система не ограничивает количество запросов
3. Не отображается сообщение о превышении лимита запросов на восстановление пароля

## Серьёзность
Major

## Приоритет
High
