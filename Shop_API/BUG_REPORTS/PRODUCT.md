# BUG_PROD_CREATE_001 (Исправлен)

## Заголовок
Залипающий ноль в поле "Цена" при создании товара (admin/seller)

## Описание
В форме создания товара поле "Цена" изначально заполнено значением по умолчанию 10. При попытке изменить это значение:
 - если стереть его, в поле автоматически появляется 0
 - если вводить новое число, когда в поле отображается 0, ноль остаётся впереди (например, 0123)

Поведение поля выглядит некорректно и сбивает пользователя с толку, несмотря на то, что при сохранении товара значение цены обрабатывается корректно.

## Предусловия
1. В системе существуют пользователи:
   - tadmin (роль admin)
   - tseller (роль seller)
2. Пользователь авторизован под логином tadmin или tseller и находится в разделе "Товары"

## Шаги воспроизведения
1. Нажать кнопку "Новый товар"
2. Обратить внимание, что поле "Цена" по умолчанию содержит значение 10
3. Полностью стереть значение в поле "Цена" клавишами Backspace/Delete.
4. Наблюдать, какое значение отображается в поле после очистки.
5. Не удаляя появившийся 0, ввести новое число, например 123.

## Ожидаемый результат
1. После очистки значения по умолчанию поле "Цена" либо:
 - остаётся пустым, либо
 - сразу принимает вводимое значение без промежуточного 0
2. При вводе нового числа значение отображается без ведущего нуля (например, 123)

## Фактический результат
1. После очистки значения 10 в поле "Цена" автоматически отображается 0
2. При вводе нового числа в поле, содержащем 0, значение отображается с ведущим нулём, например 0123
3. Визуально создаётся ощущение, что ноль является частью вводимой цены, что может вводить пользователя в заблуждение

## Серьезность
Major

## Приоритет
Medium

## Окружение
- ОС: macOS 26.2
- Браузер: Comet
- URL: http://localhost:3001

# BUG_PROD_CREATE_002 (Исправлен)

## Заголовок
Система не позволяет сохранить цену с копейками при создании/редактировании товара в разделе "Товары"

## Описание
При вводе цены товара с копейками (например, 99,89 ) UI отображает сообщение: "Введите допустимое значение. Ближайшие допустимые значения: N и K"

При этом:
- система не изменяет цену автоматически
- пользователь должен сам изменить значение (округлить до целого), иначе товар не будет сохранён

## Предусловия
1. В системе существует авторизованный пользователь tuser
2. Пользователь находится в разделе "Товары"

## Шаги воспроизведения
1. Перейти в форму создания товара
2. В поле "Наименование" ввести произвольное значение
3. В поле "Цена" ввести дробное значение, отличное от целого числа (например, 99,89 или 100,50)
4. Нажать кнопку "Сохранить"

## Ожидаемый результат
1. Система позволяет ввести цену с двумя знаками после запятой (например, 99,89)
2. UI не требует округления до целого числа
3. Цена с копейками может быть сохранена
4. В интерфейсе цена отображается с копейками (99,89)
5. Пользователь может продолжить работу без принудительного изменения цены

## Фактический результат
1. Отображается сообщение:  
   "Введите допустимое значение. Ближайшие допустимые значения: N и K"
2. Система не даёт сохранить товар, пока пользователь сам не изменит цену до допустимого (целого) значения
3. Копейки не могут быть сохранены в товар

## Серьёзность
Major

## Приоритет
High

## Окружение
- Shop App: 1.0.0
- ОС: macOS 26.2
- Браузер: Comet (сборка на основе Perplexity)
  - Версия Comet: 148.0.7778.257 (официальная сборка)
  - Номер сборки Perplexity: 1743
- URL: http://localhost:3001

# BUG_PROD_001 (Исправлен)

## Заголовок
Некорректная нумерация позиций в списке товаров после удаления элемента

## Описание
В списке товаров отображается нумерация, которая должна быть последовательной (1, 2, 3...), но после удаления товара появляются пробелы (1, 3, 4...). 
В БД ID товаров уникальны и существуют (1, 2, 3...)

## Предусловия 
1. В системе существует авторизованный пользователь t_user 
2. Пользователь находится в разделе "Товары"
3. Существует 3 последовательных товара, например:
 - Товар А с позицией = 1
 - Товар В с позицией = 2 
 - Товар С с позицией = 3

## Шаги воспроизведения
1. Обратить внимание на отображаемую нумерацию позиций 
2. Удалить Товар В (2-й в списке) через кнопку "Удалить"
3. Проверить нумерацию позиций после удаления

## Ожидаемый результат
1. Система отображает последовательные номера позиций (1, 2) 

## Фактический результат
1. После удаления товара с позицией = 2 отображается нумерация позиций: 1, 3 (видимый пробел)
2. Позиции не пересчитываются автоматически после удаления

## Рекомендации по исправлению
- UI должен отображать позицию как индекс массива + 1 (а не ID из БД)
- После удаления товара позиция должна пересчитываться автоматически
- ID в БД оставить неизменными (первичный ключ)
