## 1. Противоречия между функциональными и API-спецификациями

### 1.1 Валидация цены товара
- **REQ_PRODUCT_002/003**: минимальное значение цены = 1
- **API_PRODUCTS_004/005**: minimum = 0
- **Проблема**: API допускает цену 0, функциональные требования запрещают

### 1.2 Процесс восстановления пароля
- **REQ_AUTH_004**: требует email И логин для идентификации
- **API_USERS_002**: принимает только email (логин не передаётся в теле запроса)
- **Проблема**: функциональное требование требует два поля, API — одно

### 1.3 Логика добавления товара в корзину
- **REQ_CART_001/003**: при повторном добавлении количество УВЕЛИЧИВАЕТСЯ на добавляемое значение (инкремент)
- **API_CART_002**: количество УСТАНАВЛИВАЕТСЯ в указанное значение (set)
- **Проблема**: разная семантика операции — инкремент vs установка

### 1.4 Тип ID позиции корзины
- **API_CART_002 ответ**: `id` — integer
- **API_CART_003/004 path-параметр**: `id` — string (UUID)
- **Проблема**: несогласованные типы идентификатора в одном API

### 1.5 Функционал кошелька
- **REQ_WALLET_001**: описывает только пополнение (top-up)
- **API_WALLET_002**: поддерживает DEPOSIT и WITHDRAW
- **Проблема**: API расширяет функционал сверх того, что описано в требованиях

### 1.6 Минимальная длина пароля
- **REQ_AUTH_001** (раздел "Требования к паролю"): минимум 8 символов
- **REQ_AUTH_004** (восстановление пароля): минимум 6 символов
- **API_PROFILE_003**: минимум 6 символов
- **API_AUTH_001 ответ 400**: "минимум 8 символов"
- **Проблема**: три разных значения (6, 6, 8) в разных документах

### 1.7 Максимальная сумма корзины
- **REQ_CART_004 пункт 9**: "не может превышать 100 000 000,00 рублей"
- **REQ_CART_004 пункт 9 (продолжение)**: "при попытке добавить товар... больше 1 000 000,00 рублей"
- **Проблема**: два разных лимита в одном требовании (100 млн vs 1 млн)

### 1.8 Обработка сессий при выходе
- **REQ_AUTH_003 пункт 7**: при выходе "завершать все критичные действия, связанные с учетной записью"
- **REQ_AUTH_003 пункт 8**: "выход из системы в одной вкладке не должен разлогинивать пользователя в других вкладках"
- **Проблема**: потенциальное противоречие — какие именно действия считаются "критичными" и как это совмещается с независимостью вкладок

## 2. Ошибки в кросс-ссылках API-спецификаций

### 2.1 API_AUTH_001 (регистрация)
- Ссылается на **REQ_AUTH_002** для требований к паролю
- **Факт**: REQ_AUTH_002 — это требования к авторизации, требования к паролю находятся в **REQ_AUTH_001** (раздел "Требования к паролю")

### 2.2 API_AUTH_004 (выход)
- Ссылается на **"REQ_AUTH_004 — Политика токенов"**
- **Факт**: REQ_AUTH_004 — это восстановление пароля, политика токенов нигде не описана отдельным требованием

### 2.3 API_PRODUCTS_004 (обновление товара)
- В пункте 8 ссылается на **REQ_PRODUCT_002** для правил валидации
- **Факт**: REQ_PRODUCT_002 — создание товара, для обновления должна быть ссылка на **REQ_PRODUCT_003**

## 3. Отсутствующие функциональные требования

### 3.1 Оплата корзины (checkout)
- **API_CART_005** описывает endpoint `POST /v2/cart/checkout`
- **REQ-файл отсутствует**: нет требования, описывающего бизнес-логику оплаты корзины из кошелька

### 3.2 Очистка корзины
- **API_CART_006** описывает endpoint `DELETE /cart/clear`
- **REQ-файл отсутствует**: нет требования, описывающего полную очистку корзины

### 3.3 Версионирование API
- Checkout использует префикс `/v2/`, остальные endpoints — без версии
- **Проблема**: непоследовательность в структуре URL

## 6. Дополнительные замечания

### 6.1 API_CART_002 — неоднозначная семантика quantity
- В документе сказано: "quantity обновляется до указанного значения (или увеличивается/уменьшается, в зависимости от принятой бизнес‑логики; это нужно явно указать в общих требованиях к корзине)"
- **Проблема**: API-спецификация сама указывает на неопределённость и рекомендует уточнить в требованиях, но это не сделано

### 6.2 API_CART_003 — количество может быть 0
- API_CART_003 описывает PATCH /cart/items/{id} с quantity >= 0
- При quantity == 0 позиция удаляется
- **Вопрос**: REQ_CART_001 пункт 8 говорит "не должно позволять добавлять количество меньше 1" — противоречит ли это API?

### 6.3 REQ_CART_004 — дублирование информации
- Пункт 9 содержит два разных числа в одном абзаце (100 000 000 и 1 000 000)
- Требуется уточнение: какой лимит является верным

### 6.4 Отсутствие требований для API endpoints
- API_AUTH_003 (обновление access-токена) — нет REQ-аналога
- API_AUTH_004 (выход) — REQ_AUTH_003 есть, но описание API расширяет поведение
- API_PROFILE_001–003 — REQ_PROFILE_001–003 есть, но покрытие тест-кейсами отсутствует
